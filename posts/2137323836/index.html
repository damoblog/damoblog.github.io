沙盒类SLG战斗(四) 逻辑层总览：结构与运行流程

逻辑层里面是什么

上一章讲完了三层结构的通信机制——同帧收敛和延迟结算。

现在视角收回到三层中最核心的那层：逻辑层。

我自己第一次接触战斗系统时，会下意识地把”逻辑层”想象成一个复杂的整体：充满各种条件判断，技能逻辑和状态变化交织在一起，改一处就牵动全局，像一个难以拆解的黑盒。

等到真的做过几套不同类型的战斗系统之后，才慢慢意识到一件事：

逻辑层本身并没有想象中那么复杂——它的本质是一个 Engine 驱动多个 Manager 协同工作的系统。

复杂度不在于某一个模块本身，而在于这些模块在统一时间驱动下的协同运转。把”有哪些模块、各自负责什么、在一帧内怎么配合”这三件事想清楚，逻辑层就从黑盒变成了透明的。

模块划分：逻辑层由哪些部分组成

Engine 是整个逻辑层的驱动者——它负责战斗的时间推进、阶段切换，以及调度各个 Manager 按顺序执行。

围绕 Engine，逻辑层通常会拆出这样几个 Manager，每个只负责一个维度：

流程管理（Processor）：控制战斗整体的推进节奏。它定义一帧内各模块的执行顺序，驱动从开始到结束的阶段切换。它是整个系统的调度中心——不负责具体计算，但决定”什么时候、按什么顺序算”。

单位管理（EntityManager）：管理所有战斗实体的生命周期。单位、弹道、标记、掉落物——凡是需要在战斗里存在、消亡、被遍历的，都由它统一管理。

阵营管理（Camp）：管理敌我关系与阵营级状态。它负责回答”谁是我方、谁是敌方”，维护阵营级别的容器和属性，支持多阵营的关系扩展。

地图管理（Map）：管理空间、位置与寻路。单位出生在哪、边界在哪、怎么从 A 走到 B、攻击范围内有哪些目标——所有空间相关的问题都归它回答。

随机数管理（Random）：保证随机过程可复现。战斗里所有随机行为统一从这里取数，基于外部传入的种子生成确定序列，保证多端一致。

输入管理（Input）：收集并收敛外部输入。玩家操作、AI 决策、系统触发——各种来源的输入先在这里缓冲、排序，帧末统一交给逻辑层处理。

输出管理（Record）：记录并输出战斗事件。逻辑层产出的所有事件在这里被收集、整理成标准化事件流，提供给桥接层和表现层。

配置管理（Config）：提供技能、单位等静态数据。它解耦逻辑与数据——逻辑层通过它查技能参数和属性表，而不直接依赖具体的数据格式。

日志管理（Log）：记录关键执行过程。支持回放调试和问题定位，是系统可观测性的保障。

这些 Manager 在 Engine 的驱动下协同运作，共同完成一场战斗模拟。

一帧之内：这些模块怎么配合

光知道”有哪些模块”还不够，更重要的是：它们在一帧里是按什么顺序工作的？

一个典型逻辑帧的执行流程大致是这样的：

处理输入 → Input Manager 把本帧缓冲的所有输入排序合并，转换为逻辑可执行的指令，交给逻辑层。这一步决定了”这一帧，战斗意图是什么”。

单位执行 → EntityManager 遍历所有存活实体，驱动它们的行为逻辑（移动、攻击、AI 决策、技能释放）。这一阶段产生的伤害、属性变化、状态增减，先记录下来，不立即生效。

结算增删 → 本帧新增的实体（召唤物、弹道）和死亡的实体，在所有行为执行完之后统一加入或移除。

处理输出 → Record Manager 把本帧产出的所有事件汇总整理，统一交给桥接层。

检查胜负 → 判断胜负条件是否满足，标记战斗结束或继续下一帧。

从整体来看，一帧的执行可以抽象为：

输入 → 执行 → 结算 → 输出 → 判定

Engine 不断重复这一过程，直到战斗结束。

结语

到这里，逻辑层从”复杂黑盒”变成了一幅清晰的结构图：Engine 驱动时间，Processor 调度顺序，各个 Manager 各守一块职责，在固定的帧内按顺序协作完成战斗模拟。

接下来的章节会逐个拆解这些 Manager。

首先是那个”决定所有人什么时候动”的模块——流程管理（Processor）。
