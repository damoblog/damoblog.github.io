从复杂到简单：逻辑层的本质

在上一章中，我们从通信与同步的角度，梳理了三层结构是如何协同运转的。

也正是在这个过程中，一个结论逐渐变得清晰：

真正决定战斗结果的，是逻辑层本身。

但在我刚开始接触战斗系统的时候，其实并不是这样理解的。

那时候我会下意识地把“逻辑层”想象成一个非常复杂的整体：

  • 充满各种规则判断
  • 到处是技能逻辑与状态变化
  • 各种流程交织在一起

看起来像一个难以拆解的“黑盒”。

直到后面做过几套不同类型的战斗系统之后，才慢慢意识到一件事：

逻辑层本身并没有想象中那么复杂，它的本质，其实是一个 Engine 驱动多个 Manager 协同工作的系统。

换句话说：

  • Engine 负责驱动流程（Tick / 阶段 / 调度）
  • Manager 负责管理不同维度的状态（单位 / 地图 / 输入 / 输出等）

逻辑层的复杂度，并不来自某一个模块，而是来自：

这些模块在统一时间驱动下的协同运转

因此，在这一章中，我们不再从抽象模型出发，而是换一个更“工程”的视角来看逻辑层：

  • 它是由哪些模块组成的？
  • 这些模块分别负责什么？
  • 在一帧内，它们是如何协同工作的？

接下来，我们将从这个角度出发，对逻辑层进行一次整体拆解。

模块划分：逻辑层由哪些部分组成

在明确了逻辑层的本质之后，可以把视角从“整体”拆回到“结构”。

如果把逻辑层看作一个由 Engine 驱动的系统，那么一个更直接的问题就是：

这个系统，具体是由哪些部分组成的？

在实际工程中，我们通常不会把所有逻辑写在一起，而是按照职责进行拆分，形成一组相互协作的模块（Manager）。

这些模块各自负责不同维度的状态与规则，在统一的流程驱动下协同工作，最终完成整个战斗的模拟过程。

模块总览

各模块职责简述

流程管理（Processor）：控制战斗整体推进节奏

  • 驱动 Tick 推进
  • 控制战斗阶段（开始 / 进行 / 结束）
  • 调度各个 Manager 的执行顺序

👉 本质是：整个系统的“调度中心”

地图管理（Map）：管理空间、位置与寻路

  • 管理单位位置
  • 处理移动与空间关系
  • 提供距离、范围等查询

👉 本质是：空间信息的统一来源

阵营管理（Camp）：管理敌我关系与目标选择

  • 定义敌我关系
  • 提供目标筛选规则
  • 支持阵营扩展（中立 / 多方）

👉 本质是：战斗关系的判定中心

单位管理（Entity）：管理所有战斗单位的生命周期

  • 管理单位创建 / 销毁
  • 维护单位状态
  • 提供单位查询与遍历

👉 本质是：战斗对象的容器

随机数管理（Random）：保证随机过程可复现

  • 统一随机数来源
  • 基于种子生成随机结果
  • 保证多端一致性

👉 本质是：可复现的随机系统

输入管理（Input）：收集与收敛外部输入

  • 收集玩家 / AI 输入
  • 在帧内完成收敛
  • 按顺序提交给逻辑层

👉 本质是：战斗意图的入口

输出管理（Record）：记录并输出战斗事件

  • 收集逻辑结果
  • 组织为事件流
  • 提供给桥接层 / 表现层

👉 本质是：战斗结果的出口

配置管理（Config）：提供技能、单位等配置数据

  • 提供静态数据（技能 / 属性 / 表）
  • 支持运行时查询
  • 解耦逻辑与数据

👉 本质是：规则的数据来源

日志管理（Log）：记录过程，用于调试与回放

  • 记录关键执行过程
  • 支持回放 / Debug
  • 辅助问题定位

👉 本质是：系统可观测性保障

这些模块在 Battle Engine 的驱动下协同运作，共同完成战斗模拟。

但仅仅知道“有哪些模块”还不够，更关键的问题是：

在实际运行中，这些模块是如何在同一时间轴上协同工作的？

换句话说：

  • 输入是在什么时候被处理的？
  • 单位行为是在什么时候被执行的？
  • 数值变化是在什么时候真正生效的？
  • 事件又是在什么时候被输出的？

这些问题，最终都会收敛到一个更具体的视角：

在一个逻辑帧（Tick）内，战斗系统到底经历了哪些步骤？

接下来，我们将从“一帧”的角度出发，梳理逻辑层的完整执行流程。

一帧执行流程

在逻辑层中，所有模块的协作，最终都会收敛到一个统一的执行单位：

逻辑帧（Tick）

每一帧，Engine 都会驱动各个 Manager 按固定顺序执行，从而推进战斗状态。

一个典型的逻辑帧，大致会经历以下几个阶段：

处理输入

  • 收集当前帧内的所有输入（玩家 / AI / 系统触发）
  • 按规则进行排序与收敛
  • 转换为逻辑层可执行的指令

👉 得到：这一帧的行为意图

单位新增

  • 处理本帧需要创建的单位（召唤 / 出生等）
  • 初始化单位状态（属性 / 技能 / Buff）
  • 加入战斗系统

👉 确保：新单位能参与后续逻辑

单位执行

  • 遍历所有存活单位
  • 执行行为逻辑（普攻 / 技能 / AI 决策）
  • 记录产生的数值变化（伤害 / Buff 等）

👉 这一阶段通常只“记录变化”，而不立即生效（延迟结算）

单位移除

  • 处理本帧内死亡或离场的单位
  • 清理相关状态与引用
  • 从系统中移除

👉 保持战斗状态的正确性

处理输出

  • 汇总本帧产生的所有事件
  • 组织为标准化事件流
  • 提供给桥接层 / 表现层

👉 得到：这一帧发生了什么

检查战斗结果

  • 判断胜负条件是否达成
  • 标记战斗结束或继续下一帧

👉 控制整体流程的终止

一个整体视角

从整体来看，一帧的执行可以抽象为：

输入 → 执行 → 结算 → 输出 → 判定

而 Engine 的职责，就是不断重复这一过程，直到战斗结束。

结语

到这里，我们已经把逻辑层从“整体认知”拆解到了“运行流程”的层面：既看到了它由多个 Manager 组成的结构，也看到了在一个 Tick 内，这些模块是如何按固定顺序协同完成战斗推进的。

但真正的重点，其实还没有开始。

接下来，我们将进入更细粒度的部分，逐步拆解每一个模块的具体设计方式。

其中有些模块（例如输入收敛、单位管理、事件输出等）在不同战斗系统中的实现思路相对统一，本质差异并不大；但也有一些模块（例如战斗流程调度、数值系统、行为系统等），在不同游戏形态中会出现非常明显的设计分歧。

而这些差异，往往也是我在不同项目迭代过程中，收获变化最大的部分。

后面的内容，我会结合实际经历过的几套战斗系统设计，逐个展开每个模块的实现思路与演进过程。
