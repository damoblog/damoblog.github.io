沙盒类SLG战斗(十二) 逻辑层的辅助设施：Config 与 Log

幕后的两个角色

前面几章把逻辑层的核心模块都讲完了——从调度（Processor）到实体（EntityManager），从输入（Input）到输出（Record）。

但有两个角色一直在幕后，被其他模块反复依赖，却从来没被单独提过：

  • 技能的参数从哪来？单位的属性表存在哪？逻辑层怎么读到这些数据？
  • 战斗跑出了问题，两端结果不一致，怎么定位是从哪一帧开始分叉的？

第一个问题靠 Config，第二个靠 Log。

它们不驱动战斗流程，不参与状态计算，但没有它们，逻辑层要么读不到数据，要么出了问题无从排查。这一章把它们一起讲完。

Config：双端必须读同一份数据

战斗逻辑里到处需要查静态数据——技能的伤害系数、冷却时间、Buff 的持续时长、单位的基础属性。

这件事看起来简单，但在多端场景下有一个容易被忽视的坑。

一种常见的做法是：逻辑层定义一个 GetTable 接口，客户端和服务器各自实现——客户端从本地资源读，服务器从数据库读，只要字段名对得上就行。

看起来很灵活，但配置的一致性完全取决于两端各自的实现。客户端的表更新了、服务器的还没同步，或者两边解析逻辑有细微差异——配置就悄悄不一致了。而配置不一致的后果和随机数不一致一样：同一场战斗，两端跑出不同的结果。

所以我们的做法是：逻辑层不接受”接口 + 各端实现”，而是要求一份唯一的配置源——客户端和服务器读的是完全相同的一份数据，由外部在战斗初始化时统一传入。

配置还需要带版本号——它是双端对齐的校验手段。客户端说”我用的是 v123 的配置”，服务器也加载 v123，两边才能确认读的是同一份数据。没有版本号，配置是否一致就无从验证。

有的游戏还需要考虑 AB 测试的场景——线上可能同时存在多份配置源，不同玩家组读的是不同版本的数据。这给”唯一配置源”带来了额外的复杂度：不再是全局一份，而是按组一份，但每一组内仍然要保证双端读到的是同一份。

类似的还有兼容包的问题——线上可能存在没有更新到最新版本的老客户端，配置需要能向下兼容这些老包。新加的字段不能让老版本的解析直接崩掉，这也是版本管理需要考虑的一部分。

配置一致性的保证不能靠”两边各自实现同一个接口”，而要靠”两边读的就是同一份东西”。

从确定性的角度看，Config 和前面讲的 Input、Random 其实是同一类东西——都是逻辑层的外部输入。只不过 Input 是”动态输入”（每帧可能不同），Random 是”种子输入”（一次性传入），而 Config 是”静态输入”（整场战斗不变）。三者任何一个在双端不一致，结果都会分叉。

Config 在战斗系统里的核心职责是保证双端读到同一份数据——同一个版本、同一个源、同一套预处理规则。

Log：按帧对齐的调试手段

战斗系统里最棘手的 bug 往往不是崩溃，而是两端结果不一致——客户端和服务器跑同一场战斗，最后结果对不上。

这类问题的难点在于：你知道结果不对，但不知道是从哪一帧开始错的。一场战斗可能跑几百帧，每帧有若干单位在行动、施法、受伤——错误可能藏在任何一帧的任何一步里。

没有日志，定位基本靠猜。

Log 的设计围绕一个核心思路：让日志和逻辑帧严格对齐。

每个 tick 开头打一行分隔符，这一帧里所有的日志——伤害计算的每一步、Buff 的增减、属性的变化——都落在这个分隔符之后。

两端各自输出日志文件，按 tick 对齐做 diff，第一个出现差异的地方就是分叉的起点。日志的设计目标不是”人看得懂”，而是”能和另一端逐行对齐比较”——它本质上是一个验证工具，不只是调试工具。

确定性的四个要素

把 Config 和 Log 放回整个系列来看，一个可复现的战斗系统，本质上需要四个要素同时成立：

  • 一致的输入（Input / Config / Random）——动态操作、静态数据、随机种子，双端必须完全对齐
  • 确定的执行顺序（Processor / Engine）——同样的输入，按同样的顺序执行，才能得到同样的结果
  • 可结构化的输出（Record）——逻辑结果按帧打包，表现层才能编排
  • 可对齐的验证手段（Log）——出了问题，能逐帧、逐步比对，定位分叉起点

很多项目的战斗系统做不到确定性，问题往往不在于算法有多复杂，而在于这些约束没有同时成立——输入不统一、配置不一致、执行顺序不稳定、缺少可对齐的日志——任何一个环节松动，整个系统就失去可复现性。

确定性不是某个模块的功能，而是所有模块共同守住的一条约束。

结语

Config 和 Log 的故事很短，但它们补全了逻辑层最后的两块拼图：一个解决”数据从哪来”，一个解决”出了问题怎么查”。

到这里，逻辑层从整体结构到各个模块就基本讲完了。不同项目的战斗系统可能还会有其他辅助模块——事件总线、对象池、调试工具等等——但核心骨架大致就是这些：一个 Engine 驱动时间，一组 Manager 各守职责，在固定的帧序下协同运转，共同支撑起一个确定性的战斗模拟引擎。

第三章和第四章曾经从整体视角讲过”逻辑层是怎么被驱动的”，但那时候各个模块还没有展开。现在所有模块都讲完了，可以回到一个更完整的问题：这些模块组装好以后，整个系统到底是怎么跑起来的？谁来决定什么时候 tick、灌什么数据、拿走什么结果？
