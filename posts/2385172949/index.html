沙盒类SLG战斗(九) 可复现的随机：Random

无处不在的随机

上一章讲完 Map——回答了实体的空间维度。

战斗里还有一类东西没碰，但它无处不在——随机：

  • 暴击有概率
  • 闪避有概率
  • 掉落有概率
  • AI 选目标有时也带点随机
  • 召唤、走位、波动……几乎任何“非确定”的行为，背后都是某个随机数

这一章要回答的问题可以压成一句话：

从工程上必须保证：给定同一个种子，各端按同一份战斗逻辑、同一种消费顺序去取随机数时，第 1、2、3… 次取到的值逐次完全一致——同一种子，就是同一条序列，没有「差不多」。

回放、服务器校验、自动化仿真，本质上都是在核对这件事。所以 Random 模块的第一职责不是「看起来够乱」，而是锁死序列：少取一次、多取一次、换一行执行顺序，从那一拍起整条链就和对方对不上。

简单、直观，看起来够了。

但只要 SLG 战斗一上量，会发现这套很快撑不住——尤其是当你需要回放、服务器校验、自动化仿真这些能力的时候：它们都在做「多端对表」——第几格是什么数，必须一致；随手 math.random() 却很难给出这种硬保证。

为什么？

math.random() 为什么撑不住

math.random() 这种语言内置的随机数函数，有三类硬伤——每一条都在让「大家数的是不是同一张表」这件事落空。

一：跨平台序列对不齐

math.random() 内部用的是什么算法、什么参数？答案是——取决于实现。

  • Lua 5.1 / Lua 5.3 / LuaJIT 的 math.random 内部算法都不一样
  • C 语言的 rand() 在不同 libc 实现下也不一样
  • 即使同一份代码，在 Windows / Mac / iOS / Android 上跑结果也可能不同

也就是说：你以为固定了种子、固定了调用顺序，在不同平台上拿到的仍然可能是两条不一样的随机数序列。

回放和服务器校验要的是「第 N 次取到的数和对方完全一致」——这里一上来就输了。

二：消费顺序一乱，整条序列就废

math.random() 是个全局函数。战斗里任何模块——AI、技能、Buff、寻路、表现层——都可能在调它。

  • 表现层为了一个飘字位置抖动调一次 math.random
  • 战斗逻辑算暴击再调一次 math.random

伪随机没有魔法：RNG 只有一个游标，按调用顺序依次往后走。 多插一次、少插一次、换一行代码的执行顺序，后面所有「第 k 个随机数」全部错位。你没法跟别的进程对齐「我们约定好第 37 个随机数留给暴击」——因为根本没人记账。

三：种子不可控，序列从何谈起

math.randomseed() 也是全局的——任何模块都能改。改一次，整条序列的起点就变了。

更糟的是，你可能根本没有稳定种子：默认初始值常常是启动时间或内部噪声，每场战斗连起点都对不齐，后面更无从「对表」。

三条合起来，结论只有一句：

战斗里所有需要参与胜负、回放、校验的随机，必须从同一个、算法固定、种子明确、消费顺序可约定的 RNG 里依次取出。

这个源就是 Random。

关键决策：所有随机收敛到一个 RNG

战斗逻辑层里的设计，就是在工程上锁死「谁从 RNG 里按什么顺序取数」：

  • 一个战斗只有一个 RNG，由 Engine 持有——全战斗只有一条随机数序列
  • 由外部传入种子——服务器、战报、仿真器约定好「这一场从哪一个序列起点开始」
  • 只有逻辑层、且只有约定好的路径可以取随机——表现层、工具脚本不许偷调 math.random()，避免在序列中间插队

这样，在起点、逻辑、输入路径都对齐的前提下，各端取到的就是同一张数表上的同一串格子；客户端多跑一遍、服务器再跑一遍，数到第 k 格，数必须一样。

接口非常简洁——就两个方法：GetNext() 和 GetNextMax()。

接口背后只有一条铁律，和「帅不帅」无关：

随机数不是随手调用的副作用，而是按顺序消耗的序列资源；谁取、取几次、先取后取，都必须和别的端对齐。

算法选择：为什么是 LCG

挑随机数算法的时候有不少选项：

  • LCG（线性同余）：极简单，4 行代码，跨平台一致
  • Mersenne Twister：周期超长（2^19937），但实现复杂、状态大
  • Xorshift / xoshiro：现代算法，速度快、统计性质好
  • PCG：质量极高的现代算法

SLG 战斗里我用的是 LCG。原因不是“它最强”，而是在「同一种子 → 下一位永远算成同一个数」这件事上，它简单到很难写错：

  • 实现极简（一行核心代码：seed = (seed * A + B) % M）
  • 跨平台一致——只要算法和参数一样，任何语言、任何平台、任何 Lua 版本，同一次调用拿到的都是序列里的同一个值
  • 速度快、状态小（只有一个整数）
  • 周期足够（用 Park-Miller 经典参数，周期约 2.1 亿）——一场战斗远远用不完

战斗里要的不是密码学意义上的不可预测，而是多端对齐：同一种子下，第 1、2、3… 次取到的数必须和对方完全一致。LCG 在这件事上足够稳。

更复杂的算法（MT / xoshiro）统计性质更好，但：

  • 战斗里随机数调用频率通常没那么夸张
  • SLG 优先要的是序列对齐，不是论文里的随机性指标
  • 实现越复杂，换语言、换端移植时越容易「差一点就不一样」

这又是一次“业务驱动的选型”：

算法选择的标准不是「最先进」，而是「换语言、换端实现一遍，数格子能不能仍对得上」——够用、易移植、已验证。

战斗里其实没有”真随机”

回头看 Random 的整套设计——

  • 一个种子初始化
  • 一个全局唯一 RNG
  • 任何随机都从这里出

会发现一个有点反直觉、但对工程极其友好的事实：

战斗里其实没有「真随机」——只有「看起来像随机的确定序列」。

也就是说：起点定了、输入路径定了、取数顺序定了，整场里第几次暴击、第几次闪避、第几次掉落，早就对应到「第几个格子」上了。玩家觉得是运气，系统只是在按顺序读表。

这件事一旦想透，会发现：

SLG 里我们追求的「随机」，在工程上就是「多端能对着同一张表数下去」。

随机和确定性在这里不是对立的——它们是同一件事的两面：对玩家是运气，对系统是一张确定的数表。

这才是整个回放、校验、仿真能成立的底座。

结语

到这里 Random 的故事就讲完了。它的逻辑非常朴素：约定好起点，约定好谁能取数、按什么顺序取，各端就只是在同一张表上往后数——数乱了、表不一样，回放和校验立刻全废。

唯一 RNG、外部种子、禁止旁路，都是为了把「数表」这件事钉死。没有它，回放、服务器校验、自动化仿真这些能力都立不住。

Random 是战斗系统里最薄、但最不能省的模块之一。

到目前为止，逻辑层的“被动服务”都讲完了：

  • 实体（EntityManager）
  • 关系（Camp）
  • 空间（Map）
  • 随机（Random）

但还有一个角色没出场——外部到底怎么“驱动”战斗？玩家点了一下屏幕、AI 决定要释放技能、网络包到了——这些“外部意图”是怎么进入战斗系统的？
