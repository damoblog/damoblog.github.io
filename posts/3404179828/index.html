沙盒类SLG战斗(五) 逻辑层的调度核心：Processor

发表于2026-04-25|更新于2026-04-27|技术战斗沙盒类SLG战斗
|阅读量:
点我回到首页

在前面我们已经明确了逻辑层的基本结构：Engine + 各类 Manager 协同运作

但如果只停留在“有哪些模块”，系统其实仍然是静态的。

真正让战斗“跑起来”的，是一个核心模块：流程管理（Processor）

从更抽象的角度来看：Processor 本质上定义了战斗系统的“时间结构”

它决定了一件最关键的事情：

在一个逻辑帧（Tick）内，各个模块的执行顺序与时机。

Processor 本质在做什么？

很多人第一次接触战斗系统时，会把 Processor 理解成“流程代码”。

但在 SLG 战斗中，它更本质的作用是：

统一调度所有 Manager，在确定的时间顺序下完成一帧模拟

也就是说：

  • Processor 不负责“具体计算逻辑”
  • Processor 只负责“计算发生的顺序与时机”

为什么必须有 Processor？

如果没有统一的流程调度，会出现几个问题：

执行顺序不确定

  • 输入先处理还是单位先行动？
  • Buff 是先结算还是后结算？
  • 死亡是在什么时候清理？

如果没有 Processor，这些都会变成“实现依赖顺序”。

多模块逻辑无法对齐

  • Entity 在动
  • Input 在输入
  • Record 在产出

但如果没有统一节奏：各个系统会“各跑各的时间线”

最终导致战斗不可复现。

回放 / 服务器验证无法成立

SLG 战斗通常要求：同样输入 = 同样结果

这要求：

  • 执行顺序固定
  • Tick 内行为确定

Processor 就是保证这个确定性的关键。

Processor 的核心职责

可以把 Processor 简化成三件事：

控制战斗生命周期

  • 战斗开始
  • 战斗进行
  • 战斗结束

定义一帧的执行顺序

例如：

输入 → 执行 → 结算 → 输出 → 判定

驱动 Tick 循环

  • Engine 每一帧调用一次 Processor
  • Processor 内部推进所有 Manager

从实现上看，Processor 往往表现为一个固定顺序的 Tick 调度函数：

可以看到，所有 Manager 的执行顺序都是在这里被明确规定的，而不是由各自内部决定。

Processor 和 Manager 的关系

这点非常关键：

Processor 不“拥有逻辑”，但 Processor“决定逻辑的执行顺序”
Manager 之间不应该直接互相驱动，而是统一由 Processor 调度

  • Entity Manager 不知道什么时候执行
  • Input Manager 不知道什么时候输入
  • Record Manager 不知道什么时候输出

这些全部由 Processor 决定。

结语

回头看 Processor 这段代码，会发现一个有趣的事实：

整个 Processor 加起来只有几十行，没有状态机、没有可配置 Phase、没有动态步骤注入。

这并不是因为它“还不够完善”，而是因为：

  • 逻辑/表现的分离（二）已经把表现踢出了 Processor
  • 同帧收敛 / 延迟结算（三）已经把时间约束分摊到了输入与结算
  • Engine + Manager 的拆分（四）已经把状态管理分摊到了各个 Manager

当所有约束都被分摊到合适的位置之后，Processor 自然就只剩下“骨架”。

一个克制的 Processor，背后是被分摊得当的系统设计。

但也正是这层“骨架”，让逻辑层从一组 Manager 的集合，变成了一个可控、可复现的战斗模拟流水线。

在 Processor 的调度顺序中，最先被驱动的就是 EntityManager：所有单位的行为推进、增删、状态维护都由它负责。

下一章，我们将进入这个“被调度得最频繁”的模块——单位管理（EntityManager）。
