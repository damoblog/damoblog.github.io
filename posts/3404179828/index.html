沙盒类SLG战斗(五) 逻辑层的调度核心：Processor

|阅读量:
点我回到首页

一帧里谁先谁后

上一章讲完了逻辑层的整体结构：Engine 持有一组 Manager，每个 Manager 各守一块职责。

但”有哪些模块”只是静态的结构图——战斗真正”跑起来”，靠的不是这张图，而是一个问题的答案：

这一帧里，各个模块按什么顺序执行？

我刚开始设计战斗系统的时候，对这件事并不重视。反正 Manager 各自有 Tick，每帧调一遍不就行了？顺序嘛，大概先跑单位、再结算、再输出，随手排一下。

但只要战斗逻辑稍微复杂起来，就会发现”随手排”是行不通的。

为什么执行顺序必须被固定下来

考虑一个具体问题：同一帧内，单位行动之前还是之后处理输入？

  • 如果输入在单位行动之前处理：玩家这帧点了技能，这帧就能生效
  • 如果输入在单位行动之后处理：玩家这帧点了技能，下一帧才能生效

两种顺序都能”跑通”，但结果不同。更关键的是——一旦顺序在不同端不一致，同一场战斗就会跑出不同结果。客户端和服务器都在跑逻辑，但谁先处理输入、谁先结算伤害，哪怕差一步，后面的状态就全错位了。

类似的问题还有：

  • 单位死亡是在本帧行动之中清理，还是等所有单位都行动完才清理？
  • Buff 的到期是在结算之前判断，还是之后？
  • 新召唤的单位这一帧能行动吗？

每一个问题都有多个”看起来合理”的答案，但必须有且只有一个确定的答案——而且所有端、所有次运行都必须是同一个答案。

这就是 Processor 存在的原因：

Processor 不负责”具体计算逻辑”，它只负责把”计算发生的顺序与时机”锁死。

它是整个逻辑层的时间结构。没有它，各个 Manager 各跑各的时间线；有了它，一帧内的每一步都有确定位置，战斗才能真正可复现。

Manager 之间也不应该互相驱动。如果 EntityManager 在执行过程中直接调用 InputManager，执行顺序就分散到了各个模块内部，Processor 的作用就被绕过了。统一调度的前提，是所有人都只听 Processor 的。

初次看到这个，很容易觉得它”还不够完善”。但其实恰恰相反——它的简单是前面几章的设计决策自然积累的结果：

  • 逻辑与表现的分离（二）已经把表现层踢出了 Processor 的执行路径
  • 同帧收敛与延迟结算（三）已经把时间约束分摊到了 Input 和 Record
  • Engine + Manager 的拆分（四）已经把状态管理分摊到了各个 Manager

每一层设计把一类复杂度拿走，Processor 自然就只剩下骨架。

一个克制的 Processor，背后是被分摊得当的系统设计。

这层”骨架”看起来轻，但它真正做的事，是把一组 Manager 的集合变成了一条可控、可复现的战斗模拟流水线。

结语

Processor 的故事到这里就讲完了——它定义了战斗系统的时间结构，把执行顺序这件事从”实现依赖”变成了”显式约定”。

在它的调度顺序里，第一个被驱动的是 EntityManager。所有单位的行为推进、生灭结算、状态维护，都从那里开始。
