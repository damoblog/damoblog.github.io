沙盒类SLG战斗(十一) 战斗事实的出口：Record

点我回到首页

结果怎么出去

上一章讲完了 Input——它把外部意图统一收进来，帧末排好序，统一交给逻辑层消费。

现在来到另一端：逻辑层算完以后，结果是怎么”出去”的？

一帧里，逻辑层可能发生了很多事——A 移动了、B 释放了技能、C 掉了血、D 被击杀。这些事最终都要传到表现层，变成玩家看到的动画、特效、伤害数字。

但逻辑层不认识表现层——第二章已经把它们隔开了。逻辑层只管”发生了什么”，表现层只管”怎么演”。

那中间靠什么传递？

最直觉的想法

刚开始处理逻辑层输出时，最自然的想法还是——算完就通知。

逻辑层算完一次伤害，立刻回调表现层去扣血、飘字：

简单，响应快，写起来毫无障碍。

但只要一帧里事情多起来，这套就开始出问题。

即时通知为什么行不通

考虑一个很常见的场景：同一帧内，A 攻击了 B。

逻辑层在这一帧里依次算了这几件事：

  1. A 释放技能
  2. B 受到伤害，HP 从 100% 降到 70%
  3. 屏幕上要飘出 “-300” 的伤害数字

如果算完一步就立刻通知表现层，会发生什么？

  • 技能释放事件到了 → 表现层开始播 A 的攻击动画
  • 紧接着 HP 变化事件到了 → B 的血条立刻开始下降——但 A 的攻击动画还没播到”命中”那一帧
  • 再接着飘字事件到了 → “-300” 飘出来了，但刀还没砍到

玩家看到的效果：B 的血先掉了，A 的刀才砍过来，”-300” 飘在半空中莫名其妙。

这还只是一次攻击。如果同一帧里还有其他单位在行动——C 也在打 D、E 的 Buff 到期了——这些事件交错着送到表现层，表现层根本不知道”这一帧总共发生了几件事”，也无法做任何整体编排。

即时通知的根本问题是：表现层拿到的是一串碎片，而不是一帧的全貌。没有全貌，就无法编排。

除了表现上的混乱，还有一个更本质的问题：逻辑层直接依赖了表现层的接口。一旦逻辑层要在服务器上跑，或者脱离客户端做自动化仿真——没有表现层可调，代码直接报错。

统一抽象：IRecordElement

思路和 Input 那一章一样——把”发生了什么”从执行过程里抽出来，变成独立的、可以被传递和消费的对象。

逻辑层不再直接调表现层，而是生成一个 RecordElement，丢进收集器：

逻辑层只管往收集器里丢，不关心谁来消费——表现层、回放、仿真器，谁需要谁来取。逻辑层和表现层之间不再有任何直接依赖。

“关键表现帧”，不是关键帧

走到这里，一个自然的问题是：Record 里记的到底是什么？

不是状态快照。如果每帧把所有单位的完整状态（位置、HP、Buff 列表、CD 进度……）都存一份，数据量大，且绝大部分信息冗余——一个单位如果这帧没动、没掉血、没施法，记它的全量状态毫无意义。

Record 记的是变化事件——只有当某件”表现层需要知道的事”发生时，才生成一条记录：

我把这种记录叫做**”关键表现帧”。它不是动画意义上的”关键帧”——那是每隔固定间隔存一个完整状态快照，中间靠插值补全。关键表现帧恰好反过来：只在有事发生时，记录表现层需要知道的那个变化**。

什么该被记录、什么不该，取决于游戏想要什么表现：

  • 游戏要展示血条 → 需要 HpChange 记录
  • 游戏要飘伤害数字 → 需要 FlyWord 记录
  • 游戏要展示技能释放 → 需要 ActorCast 记录
  • 游戏要展示移动 → 需要 ActorMove 记录
  • 游戏不关心某种信息 → 就不需要对应的记录类型

Record 不是”逻辑层发生了什么就记什么”，而是”表现层需要演什么就记什么”——它是由表现需求反推出来的。

反过来说，Record 也不保证能还原完整的逻辑状态——它不记录 Buff 的内部状态、不记录 AI 的决策过程、不记录属性计算的中间数值。这些对表现层没有意义的东西，Record 里不会有。

这意味着一件容易被误解的事：Record 不是回放日志。很多人会本能地把 Record 和 replay 联系起来，觉得有了 Record 就能完整还原战斗过程。但 Record 的设计目标是”满足表现层编排”，而不是”完整还原逻辑”。真正能还原逻辑的是上一章讲的 Input——初始状态 + 输入序列 + 随机种子，重跑一遍才能得到完整的战斗过程。

Input 保证的是逻辑可复现，Record 保证的是表现可编排——它们的职责从一开始就不同。

按帧收集：TimelineRecorder

有了统一的 IRecordElement 接口，下一个问题是：这些记录怎么组织？

整场战斗跑完后，TimelineRecorder 里存的是一条时间线——每个 tick 一个格子，格子里装着那一帧发生的所有事件：

大部分 tick 可能是空的——没事发生就不记录。只有”发生了表现层需要知道的事”的 tick 才有内容，和”关键表现帧”的思路一脉相承。

这里有一个容易忽略的细节：同一 tick 内，Record 的生成顺序必须是确定的。表现层按顺序消费这些记录来编排动画，如果顺序不稳定，编排结果也会不一致。这个顺序不需要额外排序——它天然由 Processor 定义的逻辑执行顺序保证，只要逻辑执行顺序是确定的，Record 的 append 顺序就是确定的。

表现层怎么消费：批量接收与插值演绎

表现层的工作变得很清晰：每推进一个 tick，从 TimelineRecorder 里取出这个 tick 的所有记录，逐条调用 PlayElement：

关键在于：表现层拿到的是一帧的全部记录，而不是一条条碎片。

回到前面的例子——A 攻击 B，这一帧有三条记录（施法、掉血、飘字）。表现层拿到以后，可以做一件即时通知模式下做不到的事——编排。它知道这一帧”A 释放了技能、B 掉血了、要飘字了”，就能把这些事件组织成一组连贯的表现：先播 A 的攻击动画，等命中帧再触发 B 的受击和血条变化，同时飘出伤害数字。拿到的不再是碎片，而是全貌——有了全貌，才谈得上编排。而这个编排只能发生在表现层，不能由逻辑层来决定”先播什么、后播什么”——逻辑层只关心”发生了什么”，不关心”怎么演”。Record 给的是纯数据，编排是表现层自己的事。

但编排只解决了”一帧内多个事件怎么协调”。还有一层关系要理清：逻辑帧率和渲染帧率是不同的。逻辑层可能每秒 8 帧（每帧 0.125 秒），但画面是按 30fps 甚至 60fps 在渲染。

Record 给的是离散的变化事件——“A 移动到了 (5, 3)”、”B 的 HP 变到了 70%”。表现层要在两个逻辑帧之间插值，把跳变演绎成连续的视觉效果：

  • ActorMove 记录了目标位置 → 表现层在一个逻辑帧间隔（0.125 秒）内，把单位从当前位置平滑移动到目标位置——7、8 个渲染帧里匀速过去，而不是瞬移
  • ActorHpChange 记录了新的 HP 百分比 → 表现层让血条在若干渲染帧里平滑过渡，而不是从 100% 直接跳到 70%

逻辑层给的是关键表现帧——离散的、稀疏的变化事件。表现层负责在这些帧之间插值，把离散的数据演绎成连续的动画。

逻辑层的两道门

把 Record 和 Input 放在一起看：

  • Input：外部意图进来 → 先缓冲 → 帧末排序 → 统一交给逻辑层
  • Record：逻辑结果出来 → 先收集 → 按帧打包 → 统一交给表现层

它们是完全对称的——一个管”入口”，一个管”出口”，都遵循同一条原则：不即时处理，先收敛到帧边界，再统一交出去。

这不是巧合。第三章讲的”同帧收敛”，在 Input 和 Record 这里同时落了地。逻辑层被这两道门框住以后，内部就变成了一个真正封闭的确定性空间：

  • 输入通过 Input 在帧开始时进入
  • 输出通过 Record 在帧结束时出去
  • 中间的计算，外部无法干扰

Input 和 Record 一起，定义了逻辑层的边界——一个有明确入口和出口、内部完全可控的确定性计算空间。

结语

Record 的故事到这里就讲完了。它定义了逻辑层向外传递结果的统一路径：只记变化、按帧收集、批量交给消费方；表现层拿到这些”关键表现帧”之后，再通过插值把离散的事件演绎成连续的画面。

和 Input 一样，它看起来只是”缓冲 + 收集”，但它们共同做的事，是把逻辑层变成了一个边界清晰、内部完全可控的确定性沙盒——这正是回放、校验、仿真一切能力的根基。

除了这些核心模块，逻辑层还有一些辅助性的角色——比如提供静态数据的 Config、支持调试定位的 Log。它们不驱动战斗流程，但缺了谁系统都不完整。
