沙盒类SLG战斗(十九) 命令系统：延迟结算的执行容器

前面一直在提的”延迟结算”

从第三章讲三层结构的通信，到第十四章讲实体的三段式 tick，到第十五章讲属性的脏标记，再到上一章讲触发器的冷却——“延迟结算”这个词反复出现。

但始终有一个问题没有正面回答：这些”不直接执行”的操作，最终被谁收集、由谁统一执行？

答案是：命令队列。所有需要延迟结算的操作——伤害、触发器、护盾、控制——都被包装成命令对象推入队列，在 LateTick 统一消费。这一章就讲这个机制：命令系统（Command）。

不延迟会怎样

先说清楚为什么要延迟。

考虑这样一个场景：一次群体伤害技能同时打了三个敌人，每个敌人受伤后都要触发”受击时回复 5% HP”的触发器。

如果伤害计算和 HP 修改都在同一帧的同一时刻发生：

问题是：第一个敌人受伤后立刻回血，第二个敌人受伤时看到的环境和第三个敌人已经不一样了。 如果触发器里有”目标血量低于 30% 时额外触发”的条件，三个敌人的判断时机就出现了差异——第一个还没回血时去判断，第二、三个是在第一个回血之后再判断。

同帧内，所有判断应该基于同一个状态快照。修改不应该立刻生效，应该被收集起来，等这一帧的决策全部完成之后再统一执行。

这就是命令系统要解决的问题。

命令的本质：把”做什么”和”什么时候做”分开

命令模式的核心是：把操作封装成对象，延迟到合适的时机再执行。

在战斗里，”合适的时机”就是每帧的 LateTick——三段式里的最后阶段。

整个流程：

  1. Tick 期间：各个系统（技能、触发器、AI）把需要执行的操作封装成 Command，推入队列
  2. LateTick：CommandComponent 统一取出所有 Command，依次执行，执行完清空队列

没有什么复杂的调度逻辑——就是一个队列，帧末批量消费。

几个典型的命令

规则只有一条：凡是需要延迟结算的操作，就包装成 Command。 不同种类的操作对应不同的 Command 子类，但接口统一：HandleCommand(target)。这些命令本质上都是对组件的一次状态修改，只是作用的组件不同。

随着系统功能增长，Command 的种类也会跟着增长——加一种新的延迟操作，就加一个新的 Command 子类，其他地方完全不需要改动。以下是几个典型的例子：

DamageCommand：伤害

技能计算完伤害值之后，不直接调 TakeDamage，而是 new DamageCommand(...)，推入队列。等 LateTick 到来，再真正扣血。

这意味着：同一帧内无论多少伤害源命中了同一个目标，所有伤害都基于帧初的血量去计算，而不是前一个伤害扣完血之后再算下一个。

TriggerCommand：触发器的添加与移除

这个命令在上一章讲触发器时其实已经提到过——触发器不能在自身触发过程中被修改。比如”受击时添加一个新触发器”，如果直接在触发器的 TryTrig 里调 AddTrigger，就会在遍历触发器列表的过程中修改列表，产生不确定行为。

通过 TriggerCommand 延迟到 LateTick，这个问题自然消失——触发器列表的遍历已经全部结束，再统一修改列表是安全的。

ShieldCommand：护盾的添加与移除

护盾的添加和移除同样通过命令延迟——不让护盾在伤害结算中途突然生效，改变当前帧后续伤害的吸收结果。

ControlCommand：控制状态

控制命令稍微复杂一点，因为控制状态是引用计数管理的，而不是简单的开/关：

为什么是引用计数？因为同一时刻可能有多个来源的眩晕效果叠加在一个单位身上——A 技能打了眩晕 2 秒，B 技能又叠了眩晕 1 秒。如果直接用布尔值，A 效果到期把眩晕置为 false，B 效果还在，单位就意外地解控了。

引用计数解决这个问题：每加一个控制效果 +1，每移除一个 -1，只要引用计数 > 0，单位就处于控制中。refCnt 为正表示施加，为负表示解除，命令本身是统一的，不需要区分”AddControlCommand”和”RemoveControlCommand”两个类。

控制命令还内置了一个附加逻辑：当控制从”无”变为”有”时，打断当前施法——这个打断不需要单独发一条命令，直接在 ControlCommand 的 HandleCommand 里处理就够了，因为此时已经在 LateTick，当帧的施法决策已经完成。

以上只是当前系统里几个典型的命令种类。实际项目里，技能行为、属性修改、实体生灭等同样可以被包装成 Command——凡是”现在算、晚点改”的操作，都是 Command 的候选。

死亡保护

CommandComponent 的 LateTick 有一个最开始的判断：

这不是偶然的——它是处理”同帧死亡”场景的关键。

假设一个单位在某一帧的 Tick 期间被判定为死亡（HP 降到零），同帧内还有若干 Command 在队列里等着执行（比如其他技能的伤害、其他触发器的效果）。

如果不做保护，这些命令会在 LateTick 继续作用于一个已经死亡的实体，产生无意义的状态修改，甚至触发在死亡后不应该发生的逻辑（比如触发器继续触发、护盾继续添加）。

加上这个死亡判断之后：单位一旦死亡，当帧剩余的所有命令一律丢弃，不执行。死亡后的清理工作由死亡流程统一负责，不让命令队列产生干扰。

命令和三段式 tick 的对应关系

回到第十四章讲的三段式：前段（数据准备）→ 中段（决策）→ 后段（执行）。

命令系统在这个模型里的位置：

  • 前段 / 中段：各个系统（技能、触发器、AI）在 Tick 期间生产 Command——计算好结果，把操作封装成命令对象推入队列，但不执行
  • 后段（LateTick）：CommandComponent 消费 Command——依次取出所有命令，真正执行，清空队列

三段式保证了”决策”和”执行”分离；命令系统是”执行”那一段的具体载体。

Tick 期间只产生命令，LateTick 才执行命令——命令队列是延迟结算在代码层面的实体形态。

命令的统一接口带来的好处

所有命令都实现同一个接口 HandleCommand(target)，这让 CommandComponent 完全不需要知道队列里装的是什么类型的命令：

这也意味着：新增一种命令类型，不需要修改 CommandComponent。只需要定义新的 Command 子类，实现 HandleCommand，在需要的地方推入队列就行。CommandComponent 对新类型完全透明。

扩展成本是线性的——加一种新操作，写一个新 Command，只动两处地方（新类本身 + 调用方推入队列），不影响其他任何部分。

结语

命令系统的实现很简单——一个队列、一个接口、若干子类、帧末批量执行。但它承担的角色是贯穿整个战斗系统的：把”什么时候决定”和”什么时候生效”彻底分开，从而保证同一帧内的所有决策都基于同一份状态快照。 开头群伤的例子里，三个敌人的触发器判断之所以能看到一致的血量状态，就是命令延迟兜底的结果。

从第三章讲到的延迟结算原则，到第十四章落地为三段式 tick，再到属性脏标记、触发器冷却、护盾延迟生效——这些机制在概念层面都是延迟结算，而命令系统就是把这个概念兑现为可执行代码的那个容器。

实体内部的四个核心组件——属性、Attachment、AI/状态机、触发器——现在都讲完了。命令系统把它们穿成一串：触发器产生伤害命令、Attachment 产生触发器命令和护盾命令、控制效果产生控制命令，全部汇入队列，LateTick 统一落地。

接下来进入技能系统——战斗里最复杂、也最核心的部分。
