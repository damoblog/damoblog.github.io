沙盒类SLG战斗(二十) 技能系统（一）：技能到底是什么

战斗的复杂度从哪里来

前面几章逐个展开了实体的组件——属性、Attachment、AI 与状态机、触发器、命令。每个组件都有各自的设计难点，但它们加在一起，复杂度也比不上接下来要讲的这个模块。

这个模块就是技能系统。

回头看一个战斗单位，它本身其实不复杂：一组属性、一个位置、一些基础状态。真正让战斗变复杂的，是技能——每个技能都可能涉及目标筛选、伤害计算、控制效果、Attachment 施加、触发器注册、弹道飞行……而更致命的是技能之间的组合和交互：技能数量线性增长，技能之间的组合指数增长，技能和 Attachment、触发器、属性的联动让复杂度再翻一倍。

所以技能系统的架构，几乎决定了整个战斗系统的可维护上限。架构做对了，后面加再多技能都是配数据的事；架构做错了，每加一个技能都是一场局部重构。

技能系统比较大，一章讲不完。这几章会从”技能到底是什么”讲起，再展开具体的架构设计。先来看看走过的弯路。

Param 索引式配置

最早刚做战斗的时候，其实没什么成熟的思路，在网上看了一圈，发现有项目是这么做的：用一张 CSV 表配技能，每个技能是若干个 Action，每个 Action 通过 ActionType 区分逻辑，参数用 Param1、Param2、Param3 传递：

简单，上手快。但问题很快就来了：动作表里 Param1 在 Damage 里表示伤害值，在 AddBuff 里表示 Buff ID，在 Summon 里表示召唤物 ID——同一个列名，在不同 ActionType 下含义完全不同。参数越加越多，到 Param7、Param8 的时候，连写这行配置的人自己都记不清”第五个参数是什么”了。

更致命的是：所有参数都是字符串或数字，没有类型校验。配错了一个字段，编辑时毫无感知，运行时才报错——而且报的往往不是”配置错误”，而是某个计算结果异常，排查链条极长。

宽表式配置

后来经历了第一版，发现 Param 索引式的核心问题就是”参数没有名字”——参数必须有名字，不然谁都记不住。自然的改进就是：给每个参数一个有意义的列名。

可读性确实好了——每个字段都有名字，看一眼就知道填什么。但新的问题是：字段越加越多，绝大部分都用不上。 一个伤害技能不需要 SummonId，一个召唤技能不需要 DamageValue。所有动作挤在一张表里，每行有几十个空列。表越来越臃肿，改一个字段的定义可能影响到所有行——而且你不知道哪些行真的在用这个字段。

编辑器 + JSON

到这一步我意识到，CSV 不管怎么改列名，本质上还是”所有技能挤在一张表里”。要让每种 Action 有自己独立的字段定义，就需要一个技能编辑器——让每个 Action 配置自己专属的参数，而不是所有 Action 共享同一批列。

很快我进入了第二个战斗项目的实战，也第一次接触到了战斗编辑器。用可视化工具配技能，生成 JSON，每个技能是一份完整的结构化配置：

比 CSV 好太多了——参数有名字、有结构、有类型，编辑器还能做基本的校验。

但用了一段时间会发现三个问题：

  1. 复用性差：每个技能都是一份独立的 JSON，两个技能如果有相似的逻辑（比如”对目标造成伤害”），只能各自配一份，改一个忘改另一个是常事
  2. 缺乏抽象：Action 只是”数据结构”，不是”行为对象”——它描述了”做什么”，但没有统一的执行模型来规定”怎么做”
  3. 复杂技能难扩展：一旦技能需要条件判断、分支、命中后触发、死亡时触发，JSON 就会变得极其复杂，甚至陷入嵌套地狱

很多团队走到这一步觉得已经够了——毕竟有编辑器、有 JSON、有校验，看起来很正规。但本质上只是把 CSV 的复杂度搬到了 JSON 里，核心问题没有变：粒度太粗。一个技能 = 一份完整配置，你在”描述一个技能”，而不是”组合一组可复用的行为”。

代码驱动

编辑器 + JSON 解决了字段可读性和结构化的问题，但复杂技能的配置仍然难以维护。后来我尝试过代码驱动的方式——定义一个技能基类，每个技能继承它、覆写执行逻辑。

每个技能是一个类，逻辑写在代码里，灵活性拉满——什么特殊逻辑都能写。但代价也很明显：

  • 类爆炸：每个技能一个类，技能一多，类的数量快速膨胀
  • 复用率低：特殊技能不断出现，很难复用已有的技能代码——因为每个技能的逻辑都是一整块，拆不开
  • 组合性差：如果想让”火球术的伤害逻辑”用在”冰球术”上，只改元素类型不改伤害公式——做不到，只能把整段逻辑拷贝一份改
  • 复杂度失控：当多种机制叠加（伤害 + 控制 + 位移 + 触发 + 条件判断），单个技能的代码量爆炸式增长

四种方案，从配置到代码，从简单到灵活，但都撞上了同一堵墙：

问题不在于用 CSV、JSON 还是代码——而在于最小单位选错了。

它们都把”技能”当成了不可拆分的整体。只要设计的基本单元是”一个完整的技能”，就必然面临复用性差、组合性差、扩展困难的问题。我们真正需要的，是一个可组合的行为系统。

技能到底是什么

几套战斗做下来，我一直在思考技能系统到底应该怎么做。直到接触到 SLG 的战斗技能设计，才算豁然开朗。

拿几个具体的技能来拆，把描述去掉只看结构：

治疗术：选中范围内的友军 → 治疗。拆开来看——“选中范围内的友军”是筛选，”治疗”是效果。

雷击：选中攻击力最高的敌人 → 造成伤害 + 眩晕。拆开来看——“选中攻击力最高的敌人”是筛选，”造成伤害”和”眩晕”是两个效果。

火箭弹：选中区域内的敌人 → 发射弹道 → 命中后造成伤害。这里多了一层——“发射弹道”本身也是一个独立的行为，它和”造成伤害”无关，换成”命中后施加控制”也完全可以。

整理成表：

每个技能都可以拆成：谁被影响（筛选） + 怎么影响（效果）。而且不同技能之间有大量可复用的部分——治疗术和雷击的”施加效果”机制本质一样（都是修改某个属性），雷击和火箭弹都有”造成伤害”的效果，只是筛选方式不同。

一旦这样去看，视角就变了。我们把筛选叫做 Filter，把效果叫做 OnHit，那么：

  • 一个技能行为 = Filter + OnHit
  • 一个技能 = 若干技能行为 = 若干 Filter + 若干 OnHit

而且 Filter 和 OnHit 之间是多对多的关系——同一个 Filter 可以搭配不同的 OnHit，同一个 OnHit 也可以接在不同的 Filter 后面。它们不是一对一绑定的，而是可以自由组合的。

技能不是一个整体，而是一组行为的编排。

所谓”设计一个技能”，不是从头写一套逻辑，而是从一堆已有的 Filter 和 OnHit 里挑几个出来，按顺序组装。

从 Filter / OnHit 到统一的 Behavior

把技能拆成 Filter 和 OnHit 之后，再仔细看这两类东西的输入输出：

它们的共同点是什么？都是围绕同一个目标列表在工作。 Filter 往列表里填，OnHit 从列表里取——形式不同，但本质都是对目标的处理过程。

这就意味着，Filter 和 OnHit 不需要分成两套体系——它们可以统一成一个接口：

一个 Behavior 就是一个可复用的行为单元。它不知道自己属于哪个技能——它只知道”给我一组目标，我来处理”。不管是 Filter 还是 OnHit，在代码层面都是同一个接口的不同实现。

有了这个统一接口，技能就变成了一组 Behavior 的组合。比如一个”对范围内敌人造成伤害并降低防御”的技能，就是 SearchTargetFilter + AoeFilter + DamageProvider + PropertyModifier 这四个已有的 Behavior 串在一起。策划在编辑器里配技能，本质上就是在选择：用哪些 Behavior、按什么顺序、各自带什么参数。程序要做的是提供足够丰富的 Behavior 库，而不是为每个技能写一套逻辑。

这就是前面四种方案始终差的那一步——最小的单元不是技能，而是行为。 技能只是行为的容器。

好的技能系统不是能实现任何技能的系统——而是策划自己能组装出任何技能的系统。

结语

这一章回答了技能系统最核心的一个问题：技能到底是什么。

从 Param 索引到宽表、从 JSON 到代码驱动，每一步都是在尝试回答这个问题，也都因为”把技能当作一个整体”而撞墙。最终的答案是：技能不是一个整体，而是一组可复用行为的编排——系统的最小单位应该是”行为”，而不是”技能”。 Filter 负责收集和筛选目标，OnHit 负责消费目标，两者统一成 Behavior 接口——技能就是若干 Behavior 的组合。

有了这个认识，接下来的问题就很具体了：多个 Behavior 怎么串起来运转？技能在什么时机触发这些 Behavior？
