沙盒类SLG战斗(六) 单位生灭的节奏：EntityManager

EntityManager 到底管什么

上一章讲完 Processor——它把一帧内的执行顺序固定了下来。被它第一个调度的，就是 EntityManager。

但 EntityManager 到底是什么？

我刚开始做战斗系统的时候，下意识也是把它想成一个“装单位的列表”——aliveList = {} 加几个 for 循环，看起来差不多就够用了。

但只要真的写下去，就会发现这个“列表”很快就撑不住。

这一章要回答的问题很直接：

一个战斗系统里的“实体管理”模块，到底要解决什么问题？为什么必须复杂到这种程度？

实体管理在抽象什么？

在一个 SLG 战斗系统里，需要被管理的对象远不止战斗单位本身：

  • 单位（Unit / Army / Hero）—— 主战实体
  • 弹道（Missile）—— 飞行物 / 子弹 / 投射物
  • 战场标记（Marker）—— AOE 区域 / 召唤标记 / Buff 范围
  • 掉落物（Drop）—— 战利品 / 资源
  • ……

这几种东西做的事完全不一样：

  • 单位在地面行走、有 HP、有 AI
  • 弹道在天上飞、有命中判定
  • 标记是空间区域，不会“行动”
  • 掉落物只是等着被捡

但只要把视角抽到战斗流程这一层，会发现它们其实共用一组节拍：

  • 都需要 Tick
  • 都可能在某一帧加入战斗
  • 都可能在某一帧离开战斗
  • 都遵循“一帧之内、何时增、何时删、何时算”的统一约束

也就是说，它们“是什么”并不重要——真正重要的，是它们“什么时候生灭”。

EntityManager 真正抽象的，不是“单位”，而是：

生灭节奏。

它不关心实体在做什么——那是各类型自己的事。它只关心所有实体共同的生命周期节拍。

下面三节，沿着这个抽象，看 EntityManager 在设计上必须正面回答的三个问题。

设计问题一：增删能即时生效吗？

最朴素的写法是这样的：

要加就加，问题在哪？

考虑这个场景：

  • 当前 Tick，单位 A 释放一个召唤技能，召出单位 B
  • B 立刻被 table.insert 进 aliveList
  • Tick 还在遍历 aliveList，遍历到了 B
  • B 在自己刚出生的这一帧就开始 Tick、寻敌、攻击

问题就出在这：

B 在它“出生的同一帧”就行动了。

而同一帧里，其它单位的决策完全不知道 B 的存在——它们在选目标、放技能、做判断的时候，看到的“战场上有谁”里根本没有 B。但 B 已经在出手攻击它们了。

这跟 (三) 章讲的“延迟结算”是同一种问题：

  • 数值变化延迟到帧末 —— 保证攻守公平
  • 集合成员变化延迟到帧末 —— 保证遍历过程一致

这里的不变量是：

本帧加入的实体，本帧不参与遍历；它们要等到下一帧才开始登场。

更本质地说：

延迟结算不只针对数值，也针对集合本身。

设计问题二：实体类型不止一种

既然 EntityManager 的核心抽象是“生灭节奏”，一个很自然的诱惑是——

写一个统一的基类 Entity，把所有类型塞进同一个 list 不就行了？

听上去很优雅。但只要实际项目跑起来，就会撞墙：

  • Tick 顺序需要区分：单位先动、弹道后动、标记最后结算
  • 销毁逻辑各不相同：单位死亡要触发遗物掉落、弹道命中后才销毁、标记到时间过期
  • 查询语义不同：“找最近的敌方单位”不应该把弹道和标记也参与排序

也就是说：

不同实体类型共享同一种生灭节奏，但在执行顺序和处理逻辑上各有各的脾气。

代码看起来“重复”——每种类型都来一对 alive / toAdd——但这种重复是有意的：

它把“统一的生灭节奏”和“各自的语义差异”分开了：节奏由 EntityManager 统管，差异由各自类型自己负责。

EntityManager 的责任不是“装一个万能的列表”，而是：

为每种生灭节奏提供独立的轨道，并在帧内把它们统一调度。

设计问题三：互相引用怎么办？

如果我们试图在战斗开始时一次性把所有实体创建出来，会立刻撞上一个非常老的工程问题：

引用先于存在。

实体之间的引用关系常常是这样的：

  • 单位 A 出生时，要拿到自己 Commander 的引用
  • Commander 出生时，要拿到它统领的单位列表
  • Drone 出生时，要绑定它的宿主单位
  • 阵营在初始化时，要登记彼此的敌我关系

如果写一个朴素的 InitAll() 把所有实体逐个初始化：

  • A.Init 的时候，Commander 还没 Init 完 —— 引用拿到 nil
  • 换个顺序，Commander.Init 的时候 A 还没 Init 完 —— 还是 nil

任何顺序都不对——问题不在顺序，而在：

“建图节点”和“建图边”被混在一起做了。

标准的解法是把构造拆成多个阶段，依次放宽约束：

  • Init 阶段：挂载组件、完成自身数据初始化 —— 纯内部，不依赖任何其他实体
  • Born 阶段：实体”宣告自己活着”，组件完成激活 —— 可访问自身组件体系，仍不跨实体
  • Start 阶段：所有节点 Born 完毕，开始建立跨实体引用 —— 此时整个战场都已就位

三个阶段对应三条递进的约束：Init 时只有自己存在，Born 时可以访问自身组件，Start 时整个战场都已就位——约束逐步放宽，引用的时机也越来越晚。

中途出生的实体（比如技能召唤的单位），三个阶段会在同一帧内连续执行完，它在下一帧才被 alive 列表遍历到——和战斗开始时的逻辑完全对称。

这件事其实并不只发生在战斗系统里：

  • Unity 的 Awake() 与 Start() 之所以分两个回调，本质就是这个——保证所有 Awake 跑完后，互相 GetComponent 时对象都已存在
  • 依赖注入容器先 Instantiate 所有对象、再 Wire 引用 —— 同一件事
  • ECS 里 EntityCommandBuffer 在帧末统一 apply —— 同一件事

当对象之间存在引用时，构造必须分阶段：先建节点，再建边——中间可以根据需要插入更多的激活步骤。

EntityManager 的三阶段初始化，就是这条规则在战斗逻辑层的一次落地。看起来只是”三次循环”，但它解决的问题，比这三次循环看起来重要得多。

一帧之内：EntityManager 的执行节奏

把上面三件事拼起来，一个合格的 EntityManager 在一帧里的标准节奏就清楚了：

这个顺序背后藏着一条不变量：

本帧产生的“集合变化”，本帧之内不可见；它们要等到下一帧才被遍历到。

这条不变量保护的是什么？

  • 召唤的单位不能“凭空打人”
  • 死亡的单位不能“在死后还参与一次结算”
  • 同一帧内，所有遍历到的实体看到的“战场总览”是一致的

看上去只是几个 toAdd 队列加几个 Check 函数——但它真正撑起的，是整个战斗系统“一帧之内大家在同一时间点被评估”的公平性承诺。

“庞杂”不是设计失败

回头看 EntityManager——它要管多种实体类型、维护 alive / toAdd / dead 多组列表、处理三阶段初始化、在帧末统一结算增删。从代码量上看，它通常是逻辑层里”看起来最不优雅”的模块之一。

但它做的事其实是：

  • 把多种语义完全不同的实体抽象成同一种生灭节奏
  • 把”集合的变化”也纳入了延迟结算的不变量
  • 把”互相引用”这个工程老问题拆成了 Init / Born / Start 三阶段

EntityManager 真正抽象的不是“单位”，是“生灭节奏”。

它的“庞杂”，不是设计失败的痕迹，而是它把这些问题真的处理掉之后留下的样子。

结语

到这里，EntityManager 的故事就讲完了——它解决的是实体本身的问题：什么时候出生、什么时候死亡、什么时候被遍历。

但它只解决了“实体的存在”，还没有解决“实体之间的关系”：

  • 谁是我方、谁是敌方？
  • 攻击时该选哪个目标？
  • 战斗胜负又如何判定？

这些问题不在 EntityManager 的职责范围内——由它的兄弟模块 Camp（阵营管理） 接手，也是下一章的主题。

下一章会看到——一旦“实体”和“实体关系”被正确分开，原本看着复杂的目标筛选、敌我判定、胜负判断这些事，都会变得意外地简单。
