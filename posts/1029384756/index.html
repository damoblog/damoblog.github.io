为什么需要战斗校验

客户端算出来的战斗结果，服务器凭什么信？

客户端可能有 Bug，可能被外挂篡改，不同平台还可能算出不一样的数。服务器要想确认结果是对的，只有一个办法：自己再算一遍。

这也是前面反复强调确定性的原因——给定相同的起点和输入，不管谁来跑，结果都得一样。否则服务器算出来和客户端不一样，校验就废了。

两种战斗，两种校验思路

SLG 里的战斗大致分两类。

第一类是实时战斗——有玩家操作，释放技能、切换目标，战斗过程不可预知。服务器没法提前算完，必须等客户端跑完、把过程数据上报后再验。校验的方式是”重跑”——服务器拿着客户端上报的初始状态、种子、输入序列，自己从头算一遍。

第二类是回放战斗——没有实时操作，阵容一确定，结果就确定了。PVE 扫荡、自动 PVP 都属于这类。服务器可以先跑完整场战斗，客户端只负责播放结果。校验的压力更小，但对表现一致性要求更高——因为玩家会反复看战报，播出来的结果如果和服务器判的不一样，舆情压力很大。

不管哪种，底层面对的确定性问题是一样的。随机数、输入序列、逻辑定帧前面各有一章讲过。这里重点说一个前面没展开的问题——浮点数精度。

浮点数

我第一次遇到这个问题是在做多端校验的时候。同一场战斗，客户端和服务器跑出来的结果不一样——差了一个兵的存活。排查了很久，最终定位到一个伤害计算：firePower * 1.5 * defenseCoeff，这行代码在 x86 上算出来是 299.99999…，在 ARM 上算出来是 300.00000…。一个 floor 下去，一边是 299 一边是 300——差了 1 点伤害，传播到后面，最终差了一个兵。

为什么会这样？

  • x86 的浮点单元（FPU）默认用 80 位精度做中间计算，ARM 用 64 位——精度不同，结果就可能不同
  • 编译器可能把 a * b + c 优化成一条 FMA 指令，浮点不满足结合律，运算顺序变了结果就变了
  • 不同版本的 Lua 虚拟机对 number 的处理也可能有差异

差异极小，但战斗逐帧累积，几百帧下来就不是小事了。

我们的做法是定点数——把所有小数乘以一个固定倍率（比如 10000），全部转成整数运算。整数在任何平台上都是精确的。

除了基本的四则运算，还有几个坑：

三角函数：sin、cos 在不同平台上的实现不同，同一个角度可能算出微小差异的结果。做法是查表——预先算好一张角度到值的映射表，运行时直接查，不调系统函数。精度取决于表的粒度，比如每 0.1 度存一个值，寻路和技能范围判断够用了。

开方：sqrt 也有平台差异。用牛顿迭代法之类的可控算法替代，保证每一步计算都是确定的整数运算。

连续运算拆分：a * b * c 这种连续乘法，中间结果在 Lua 的 number 里可能超出整数精度范围（Lua 5.3 之前 number 就是 double，整数部分只有 53 位）。拆成分步运算，每步都 floor 一次：

说实话，定点数写起来比浮点数麻烦不少——scale 选多大要设计（太小精度不够，太大容易溢出），每步都要手动 floor，debug 的时候满屏都是几百万的整数很不直观。

我踩过一个很坑的问题：两个定点数相乘的时候，如果都带着缩放因子，a * b 的结果会是原来的 10000² 倍——直接溢出变成 0 了，表现就是所有伤害都是 0，排查了半天才意识到。正确的做法是先把其中一个除掉缩放因子再乘：math.floor(a * FIXED_SCALE_INV * b)，或者拆成分步运算。这种坑踩一次就记住了，但定点数的代码确实比浮点数要多操不少心。

这些都是为确定性付出的工程代价。浮点数在单一平台上可能比定点数精确得多，但定点数在所有平台上算出来都一样——校验要的就是这个。

实时战斗怎么校验

实时战斗的校验流程很直接：客户端跑战斗的同时，桥接层记录初始状态（阵容、属性、种子）和每帧的输入。战斗结束后连同结果一起上报。服务器拿同一套逻辑代码，用上报的数据从头跑一遍，比对结果。一致就过，不一致就标异常。

服务器跑的代码和客户端必须完全一样。这也是逻辑层必须和表现层隔离的原因——服务器上没有渲染没有动画，只跑纯逻辑就行。

实际项目中，”不一致”不一定直接判作弊。定点数没做干净、版本更新残留、热更新时序差异，都可能导致合法的不一致。通常会设一个容差范围——比如双方结果的兵力差异在 N% 以内算通过，超出才标异常。完全零容差听起来理想，但在线上环境里容易误判。

还有一个容易忽略的点：校验失败后怎么处理？直接回滚玩家收益？还是先记日志、积累到一定频率再介入？我们的做法是分级——偶发的不一致只记录不处理（可能是代码刚改了还没覆盖到），频繁的不一致触发告警，确认是作弊的才做处罚。校验是防线，但防线也需要容错。

回放战斗怎么播

回放战斗是服务器先跑完的，客户端只需要”看”。问题是客户端怎么看？

做法一：客户端重跑。 客户端拿到初始状态和种子，自己跑一遍逻辑，边跑边播表现。数据量小、流量少，但有个风险——如果客户端和服务器的代码版本有差异（哪怕只是改了一个公式），结果可能对不上。更麻烦的是旧版本的战报，代码都改了，重跑出来的结果和当时不一样。

做法二：客户端播录像。 服务器跑完后把 Record（表现数据）发给客户端，客户端不跑逻辑，直接播——和看录像一样。100% 和服务器一致，旧战报也能播，但数据量大、流量高。

两种做法各有适用场景，很多项目会根据玩法选择——实时性敏感的用重跑，一致性要求高的用录像。我经历的项目里两种都用过——PVE 用重跑加战力比对，数据量小、加载快，校验靠结果对比兜底；PVP 用录像，因为玩家对 PVP 结果更敏感，一旦客户端播出来的和服务器判的不一样，舆情压力很大，录像能 100% 避免这种问题。

录像数据怎么压

选了录像方案，数据量就成了问题。Record 那章已经定了基调——只记关键表现帧，不记全量状态。在这个基础上还有几招：

数值离散化：逻辑层用定点数算到很高的精度，但表现层不需要这么细。比如血条在屏幕上撑死就几百个像素，HP 从 10000 变成 9999 和从 10000 变成 9998，血条上看不出任何区别。所以 HP 变化只同步一个百分比（乘 255 取整就够了），不需要传原始的定点数值。飘字也一样——逻辑层算出来的伤害是 1234567（定点数），飘字上显示 123 就行了，玩家不需要看到小数点后的精度。

参数化：有些东西是逐帧连续变化的——CD 条倒计时、单位从 A 点移动到 B 点、血条平滑下降。如果每帧记一个值，数据量会很大。其实只需要记几个参数：CD 条记开始时间和总时长，移动记起点、终点和速度，血条记起始百分比和目标百分比。表现层拿到这些参数后自己插值，每渲染帧算出当前值——效果和逐帧记录完全一样，数据量小得多。

逻辑和表现分离：弹道在逻辑层只是一个”从 A 到 B，飞行 N 帧后命中”的数据，怎么飞、飞什么弧线、有没有拖尾特效，全是表现层的事。飘字也是——逻辑层只给出”伤害 123，暴击”，飘字飘在哪个位置、用什么颜色、向哪个方向漂移，表现层自己决定。Record 记的不是”弹道在第 3 帧飞到了坐标 (5.2, 3.1)”，而是”弹道发射了，目标是 B，飞行时间 4 帧”——剩下的表现层自己补全。

压缩的思路不是”少记”，而是”只记表现层自己推算不出来的”。

结语

战斗校验不是一个单独的功能。三层隔离、帧收敛、唯一 RNG、定点数——前面讲的每一个设计决策，最终都指向同一个问题：服务器跑一遍，能不能得到一样的结果。确定性不是一个模块，而是一条贯穿所有模块的约束。打破它只需要一行代码——一个不该调的 math.random，一个漏了 floor 的浮点乘法——守住它需要整套架构的配合。

到这里，逻辑层的内容基本就结束了。从三层架构到模块拆分，从实体组件到技能系统，从确定性保证到战斗校验——有了这些，战斗系统的骨骼算是立起来了。但骨骼只是骨骼，玩家真正看到的是表现层的东西——动画怎么播、特效怎么触发、血条怎么跟、镜头怎么跑。
