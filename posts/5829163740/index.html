沙盒类SLG战斗(三十) 弹道、飘字与特效：纯表现的模块

纯表现的模块

前面讲的实体、组件、插值，都是表现层的”基础设施”。这一章讲几个具体的表现层模块——弹道、飘字和特效。它们有一个共同的特点：完全不影响战斗逻辑，但直接决定玩家对战斗”好不好玩”的判断。

弹道：命中是逻辑决定的，飞法是表现决定的

逻辑层的弹道复杂程度因游戏而异——有些需要精确碰撞、追踪、分裂，逻辑层会相应地做得很重；有些只需要”从施法者到目标，飞行 N 帧后命中”，逻辑层仅记录飞行时长，没有任何视觉属性。

但无论哪种情况，表现层都必须自己演绎飞行过程。逻辑层不可能每帧同步弹道位置，Record 只在关键时刻发出（发射、命中），中间的飞行轨迹由表现层自己推算。推算方式有两种：一种是独立实现一套视觉曲线，和逻辑层算法无关；另一种是在表现层复现同一套算法，让视觉和逻辑在数学上保持一致。选哪种取决于这款游戏的弹道是否需要视觉和逻辑精确对齐。

表现层为弹道设计了多种飞行方式：

直线弹道：从发射点直飞目标，速度恒定，轨迹笔直。适合枪械类武器。

抛物线弹道：经过一条抛物线落到目标点，有高度变化，看起来有”重量感”。适合炮击类技能。

惯性弹道：带初始方向和速度，飞行中逐渐转向目标——先往前飞一段，再拐过来。看起来像导弹。

连接弹道：不是飞行物，而是施法者和目标之间的持续视觉连接——闪电链、治疗光束。没有飞行时间，出现即连接。

同一个逻辑弹道，配成直线是子弹，配成抛物线是炮弹，配成惯性是导弹——飞行方式、模型、拖尾、命中特效全部由表现层的配置决定，逻辑层完全不知道。命中是逻辑决定的，飞法是表现决定的。

飘字：数字背后的信息设计

飘字看起来简单——数字浮出来，往上飘，淡出消失。但实际做起来有不少细节。

飘字的类型：不只是伤害数字。普通攻击、技能伤害、暴击、治疗、护盾吸收、闪避——每种用不同的颜色、大小、动画来区分，玩家一眼就能看出”这一下是暴击”还是”被护盾挡了”。

飘字的位置：不能都从单位中心飘出来，否则多个飘字会叠在一起。通常加随机偏移，或按顺序错开——第一个偏左，第二个偏右，第三个偏上。

飘字的生命周期：飘出来 → 上浮 → 淡出 → 销毁。每个阶段的时长和曲线可以配置。飘字对象一般用对象池管理，避免频繁创建和销毁。

飘字的数据来自 FlyWordRecord，由逻辑层在 DamageProvider 里产出，包含数值、类型、目标 ID。表现层收到后创建飘字对象，设置样式，播放动画——逻辑层只管”打了多少伤害”，”用红色大字弹出来”是表现层自己的决定。

特效：粒子与时间轴

战斗里的特效分两大类：

粒子特效：一个粒子系统实例——火焰、烟雾、光圈、爆炸。挂在某个位置或跟随某个单位，播放一次或循环，到时间自动销毁。

时间轴特效：一个预制体，里面编排好了多个元素（粒子、模型、动画、音效）的播放时序。比如一个技能释放特效：”先闪光 → 再出现法阵 → 最后爆炸”，各元素的时序在编辑器里预先配好，播放时按时序推进。

特效的触发来源有三种：

  • Record 驱动：技能释放、命中、死亡——由对应的 RecordElement 在 PlayElement 时创建
  • 弹道关联：枪口火光、飞行拖尾、命中爆炸——和弹道的生命周期绑定
  • 状态关联：眩晕时头顶冒星星、Attachment 期间身上的光环——和 StateComponent 的状态变化绑定

特效管理的关键是生命周期——创建了就要确保销毁。跟随的目标死亡或被移除时，特效不能跟着悬空，否则会挂在一个已销毁的对象上导致报错。边界情况要在特效系统里统一处理，不能依赖调用方。

三者背后的同一个模式

弹道、飘字、特效看起来是三个独立的模块，但背后是同一套设计：

  1. 数据来自 Record——逻辑层产出发射、飘字、触发特效的 Record
  2. 创建由 System 管理——MissileSystem、FlyWordSystem、EffectSystem 分别管理各自的实例
  3. 每帧由 System 统一 Update——更新所有活跃实例的位置、状态、动画
  4. 生命周期自管理——弹道命中后销毁，飘字淡出后销毁，特效播完后销毁

这和逻辑层的 Manager 模式如出一辙。本质上，弹道、飘字、特效都是短生命周期的表现实体，由各自的 System 统一托管——和前面的实体/组件体系是同一套设计理念在不同粒度上的复用。

结语

逻辑层只管”发生了什么”，表现层管”怎么好看”——弹道飞什么弧线、飘字用什么颜色、特效放几个粒子，逻辑层一概不知。

逻辑层追求确定性和正确性，表现层追求体验和美感。两者各自优化，互不牵扯。 这是整套三层架构在具体模块上最直接的体现。
