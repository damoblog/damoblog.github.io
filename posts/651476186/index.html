沙盒类SLG战斗(七) 阵营本身也是实体：Camp

|阅读量:
点我回到首页

不属于任何单位的那些问题

上一章讲完 EntityManager——它解决的是“实体的存在”。

但战斗里还有一类问题，不属于任何具体实体，而是属于整个阵营：

  • 整个阵营的朝向、总血量、资源池是多少？
  • 阵营级技能怎么调度？
  • 多阵营关系（中立 / 同盟 / 三方）怎么管？
  • 战斗胜负怎么判？

里面最简单的一条是“谁是我方、谁是敌方”——给每个单位挂一个 campIndex 字段就够了：

比较两个单位的 campIndex 就能判敌我。这一招直到今天，我做的战斗里也一直保留着——简单、直接、代价低。

但 campIndex 只是一个标签——除了判敌，剩下那些问题它一个也答不了：

  • 不能挂状态（朝向 / 总血量 / 资源池）
  • 不能 Tick（阵营级延迟技能、阵营级触发器）
  • 不能管容器（这个阵营有哪些单位 / 标记 / 掉落物）
  • 不能描述阵营之间的复杂关系（中立、联合、多方混战）

这些阵营级的管理需求，需要一个真正的“阵营”概念来承载。

这一章就来回答：这个“阵营”概念应该长成什么样。

阵营级管理要解决的问题

阵营级管理具体要管什么？下面四件事，全是 campIndex 解决不了、但战斗里离不开的。

阵营有自己的状态

考虑一种 SLG 战斗里非常常见的设定：

“攻方阵营整体朝右进攻，守方阵营整体朝左防守。”

这个“前向”是单位 AI 的决策依据——“向前移动”、“转向敌人”这些行为都要用到它。但它属于谁？

它显然不属于某个具体单位——所有单位共享的就是同一个朝向。

挂在某一个单位身上不对（其它单位读不到）；挂在所有单位身上每人一份也不对（数据冗余，阵营变向时还得遍历所有人改一遍）。

类似的还有：

  • 阵营级总血量（多兵种共享一个总 HP 池）
  • 阵营级资源池（金币 / 法力 / 行动点）
  • 阵营级胜负条件参数（“X 时间内消灭所有敌人则胜”）

这些状态都是阵营级别的，不属于任何具体单位。它们必须有一个地方安放。

阵营有自己的生命周期

阵营的生命周期跟实体本身的生命周期是平行的，而且先于单位被创建出来。

战斗开始时，顺序是这样的：

  1. 先创建阵营——把 campA / campB 这些阵营本身先 Born 出来，带上自己的属性数据（初始朝向、起始资源等）
  2. 再创建单位——每个单位出生时绑定到对应阵营（写入 campIndex，同时把自己加进 camp.units[]）
  3. 建立阵营之间的关系——把对方阵营的单位登记进 camp.enemies[]，把阵营关系表填好

也就是说：

没有阵营，就没有“单位的归属”——单位必须先有阵营可绑，才能被合理地创建出来。

战斗进行中：

  • 每个阵营追踪自己单位的存活情况
  • 每个阵营响应单位死亡 / 撤退等事件

战斗结束时：

  • 每个阵营自己回答“我赢了吗”
  • 每个阵营清理自己的资源 / 容器 / 关系

这是一个完整的“出生 → 推进 → 销毁”流程，跟实体本身的生命周期平行。而且阵营的生命周期严格包住单位的生命周期——阵营先于单位创建、晚于单位销毁。

阵营有自己的 Tick

战斗里有些行为只属于阵营，不属于任何具体单位：

  • 阵营级的延迟技能调度
  • 阵营级的召唤者（不归属于任何具体单位的召唤源）
  • 阵营级的事件触发（比如“我方第三个单位死亡时触发反击”）

这些行为都需要一个 Tick 入口来推进。挂在任何一个具体单位上都别扭——因为它们本来就不属于谁。

阵营之间的关系

“两个 campIndex”能告诉你两个单位是不是同一阵营，但说不清两个阵营之间是什么关系。SLG 里常见的需求是：

  • 中立阵营（不主动攻击谁，被打才反击）
  • 三方混战（三方互为敌方）
  • 联合阵营（A 和 B 同盟，共同对抗 C）

要描述这些关系，需要一张阵营 vs 阵营的关系表——它不属于任何一个具体单位，必须挂在阵营层面。

把这四件事放一起看：

“阵营”不只是一个标签——它有自己的状态、自己的生命周期、自己的 Tick、自己的关系网。

campIndex 仍然在做它该做的事——单点判敌；但这四件事远在它的能力之外，需要一个真正的“阵营”概念来承担。

关键转折：Camp 也是一种实体

想清楚这件事之后，下一步的设计就自然了：

既然阵营有状态、有生命周期、有 Tick……那它跟一个实体还有什么区别？

答案是——没有区别。

Camp 不是“管理实体的中心”，它本身就是一种特殊的实体。

具体落地，就是让 Camp 直接继承自实体基类：

一旦做了这层抽象，好处是连锁的：

  • Camp 自动复用实体的属性系统——阵营级状态（朝向 / 总血量 / 资源池）直接挂上去
  • Camp 自动拥有 Born / Start / Tick / Destroy——生命周期完整复用
  • Camp 自动能被 EntityManager 当成一种特殊实体来管
  • 后续要加阵营级行为时，不用再发明新的容器和调度机制

这是继承用得很漂亮的一个例子，背后是一条更普适的设计经验：

当一个“管理者”开始拥有自己的属性、生命周期和 Tick 时——它已经不再是管理者了，它就是一个实体。

这个抽象的延伸性也很强。SLG 业务复杂时，往往还会在 Camp 之下再分层——比如 Legion（军团）、Squad（分队）——同一阵营下的不同军团各自有属性、有指挥逻辑、有内部 Tick。这些层级和 Camp 用的是同一种思路：每一层都被实现成实体的子类，复用同一套机制。

也就是说——

业务需要在哪个粒度上做“群体管理”，就在哪个粒度上立一个实体。

Camp 在战斗里实际承担的职责

把 Camp 当成实体之后，它在战斗里承担起几个维度的职责。

维度一：按阵营分组的容器

EntityManager 是按“实体类型”分组（aliveUnits / aliveMissiles / aliveMarkers），Camp 是按“阵营”分组：

同一个单位实体，会同时出现在两个地方：

  • EntityManager 的 aliveUnits 列表里（按类型遍历）
  • Camp 的 units 列表里（按阵营遍历）

这两组列表互不冲突——它们从两个不同的角度索引同一组实体。

维度二：显式的敌方列表

campIndex 解决的是单点判敌——给定一个目标，比较 campIndex 立刻知道是不是敌人，O(1)。

但很多查询不是单点的，是批量的：

“把这个阵营所有还活着的敌人都列出来。”

这种查询走 campIndex 就是 O(n)——每次都要扫一遍所有 alive 实体过一遍。

Camp 把这种批量查询也改成了 O(1)：

战斗开始时，每个 Camp 把对方阵营的所有单位都 AddEnemy() 一次性加进自己的 enemies[]。之后所有“列出敌人”的查询都直接读这个列表。

更本质的一句话是：

敌人列表不是查询出来的，是登记进来的。

campIndex 和 Camp 的 enemies[] 各管各的：单点判敌走前者，批量列出走后者。两者不是替代关系，而是分工。

维度三：阵营级属性

因为 Camp 继承自实体基类，它直接复用同一套属性系统，可以挂上：

  • 阵营朝向（攻方朝右、守方朝左）
  • 阵营级总血量
  • 阵营级资源池
  • 阵营级胜负条件参数
  • 是否手动放技能（玩家手动操控 / AI 自动调度，全阵营共用同一个开关）

最后一条值得单独说一句：一个阵营里可能有多个单位，但”现在是手动模式还是自动模式”这个状态只有一份——它属于阵营整体，不属于任何具体单位。把它挂在 Camp 上，正是因为它的语义本来就是阵营级别的。

这些都挂在 Camp 这个实体上，跟挂在普通实体上没有任何区别。属性系统不需要为”阵营”做任何特殊处理。

维度四：阵营级行为（Tick + 胜负）

Camp 有自己的 Tick 入口，用来推进：

  • 阵营级召唤者
  • 阵营级的延迟技能调度
  • 阵营级的延迟事件

胜负判定也是阵营级的：

注意一个细节——胜负判定不是全局函数，而是每个 Camp 自己回答“我赢了吗”。这让多阵营战斗（中立 / 三方）天然支持：每个阵营按自己的规则回答自己的胜负，全局只是把所有阵营的回答汇总一下。

两个正交的索引维度

回头看 EntityManager 和 Camp，会发现它们其实是同一组实体的两个正交索引：

每个具体实体（比如 unit A1）都被两个维度同时装着：

  • EntityManager 用它做“按类型 Tick”
  • Camp 用它做“按阵营查询 / 关系判定”

EntityManager 按类型切分实体，Camp 按阵营切分实体。同一个实体被两个维度装着，从两个角度被遍历。

这是一种稀疏矩阵式的双索引设计——不是冗余，而是不同查询语义的最优解：

  • “推进所有实体的行为” —— 走 EntityManager
  • “查出某个阵营还活着的所有单位” —— 走 Camp

硬把两件事合并到一个数据结构里，反而会让两边的代码都难写。

结语

到这里 Camp 的故事就讲完了。它和 EntityManager 是逻辑层里一对互补的模块：

  • EntityManager 解决“实体的存在”
  • Camp 解决“实体的关系”

如果只用一句话总结这一章：

实体的关系，归根到底也是一种实体。

Camp 继承自实体基类，让“阵营”这个抽象自动获得了所有实体的能力——属性、生命周期、Tick。“管理者本身也是被管理对象”的这种递归抽象，在战斗系统里其实随处可见，只是在 Camp 这一层最为清晰。

到目前为止，逻辑层已经回答了两个核心问题：

  • 实体本身是什么样的？（EntityManager）
  • 实体之间如何组织？（Camp）

但实体不是悬在空中的——它们存在于一张地图上，有位置、有距离、有视野范围。

它们的“空间维度”，谁来管？
