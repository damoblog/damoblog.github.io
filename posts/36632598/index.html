沙盒类SLG战斗(八) 实体的空间维度：Map

点我回到首页

还缺一个维度：位置

上一章讲完 Camp——回答了“实体之间的关系”。

但实体还有一个维度没碰：位置。

战斗里跟“位置”相关的问题其实非常多：

  • 单位出生时放在哪？
  • 召唤物出现在哪？
  • 掉落物落在哪？
  • 子弹能飞多远？飞到边界外是不是该消失？
  • 谁在我的攻击范围内？
  • 从这里到那里怎么走？

这些问题如果散落在各个模块里各自处理，很快就会乱——它们需要一个统一的“空间数据中心”来回答。

这个模块就是 Map。

它不是表现层那张可视化的“地图”，而是逻辑层的空间数据中心——它管的是“战斗机制里的位置”，不是“画面上的位置”。

第一个关键决策：坐标系必须分开

我自己刚开始做战斗系统时，下意识地用 Unity 的世界坐标当作“战斗里的位置”——直接用 transform.position 来算距离、判定范围。

这是一个会很快撑不住的设计。

为什么？因为逻辑层的“位置”和表现层的“位置”，根本是两件事：

维度 逻辑层 表现层
用途 战斗结果计算 视觉呈现
精度 定点数（确定性要求） 浮点数（性能 + 视觉）
频率 按 Tick 推进（如 8Hz） 按渲染帧推进（如 60Hz）
维度 战斗机制需要的最少维度 引擎天然的三维
单位 抽象数值（“1 个单位距离”） 具体显示（米 / 像素 / 屏幕坐标）
依赖 不依赖任何渲染引擎 依赖渲染引擎

如果两者绑在一起，会立刻撞上几个问题：

  • 浮点精度差异 → 不同设备结果不一致 → 战斗不可复现
  • 渲染帧率波动 → 战斗节奏被表现层拖着走
  • 服务器跑战斗 / 回放 / 自动化仿真都拿不到 transform，逻辑层根本跑不起来
  • 表现层调整（关卡缩放 / 单位大小 / 视角）会反过来影响战斗结果
  • 逻辑组和表现组的开发节奏被绑在一起，谁都没法独立推进

把这几个问题放一起看，分离的真正动机就清楚了——

不只是为了“精度对齐”，更是为了让逻辑层能独立活下来：脱离表现层、脱离渲染引擎、脱离任何具体单位（米 / 像素），只用抽象数值描述战斗里的关系。

所以 Map 设计上有一条最底层的原则：

逻辑层和表现层的坐标系必须分开——逻辑层只关心“单位之间的距离是 1 还是 2”，表现层负责把这个数值映射到屏幕上看起来多大。

具体落地：

同样的逻辑层数据，不同的关卡可以配不同的 VIEW_SCALE——一关看起来宏大、一关看起来紧凑——但战斗结果完全一致。这就是分离带来的最大收益：表现自由调，逻辑不受影响。

维度也要按需取

有一个很容易踩的坑——因为引擎是 3D 的，就把逻辑层也搞成 3D。

实际上：

  • 大多数 SLG 战斗 → 逻辑层用 xy 平面就够了，单位都在地面，z 轴只是表现层用来做高低差的视觉效果
  • 飞行单位 / 海空联合 / 立体阵地的战斗 → 逻辑层可能需要带 z 轴，因为高度会影响战斗结果（对空射程、视野遮挡）
  • 卡牌战斗 → 逻辑层连 xy 都不需要，位置是“前排第 N 号位”这种抽象槽位

判断标准只有一条：

逻辑层的维度 = 战斗机制实际需要的维度。不要因为“看起来是 3D”就把逻辑层也搞成 3D。

Map 的核心职责

坐标系明确之后，Map 在战斗里要回答几类具体的问题。

职责一：决定出生位置

战斗里有大量“要在地图上放一个东西”的需求：

  • 战斗开始时，每个阵营的每个阵位放在哪？
  • 召唤物从哪里出现？（召唤者位置 / 阵营出生区 / 指定坐标）
  • 掉落物落在哪？
  • 特殊单位（Boss / 增援）的入场点在哪？

这些位置都不应该让单位自己决定——它们是关卡配置 + 阵营 + 阵位编号的查询结果，归 Map 管：

把“出生位置”这件事统一交给 Map，单位、召唤物、掉落物的代码都不用关心“地图长什么样”——只需要告诉 Map“我要哪种位置”。

职责二：决定空间边界

战斗里很多东西需要被“约束在地图里”：

  • 单位移动不能走出地图
  • 子弹飞到边界外就消失
  • 掉落物只能落在某个安全区域内
  • AOE 范围超出地图就被截断

这些都是 Map 提供的边界判定：

边界判定看起来朴素，但它是整个战斗保持“封闭”的前提——没有它，子弹会飞去无穷远，单位会走到坐标 NaN，战斗就乱了。

职责三：寻路

“从 A 到 B 怎么走”听起来是个简单问题，但不同战斗类型下，寻路的复杂度天差地别：

  • 预设路径型（简单 SLG / 塔防）：策划在地图上配几条 path，单位按 path 走，Map 只需要返回 path 上的关键点
  • 栅格寻路型（中型 SLG / 战棋）：地图离散成格子，Map 提供 A* on grid
  • NavMesh 寻路型（RTS / MOBA）：地图是连续平面，Map 提供 NavMesh 寻路 + 局部避障
  • 3D 寻路型（沙盒 / 开放世界）：3D 空间 + 物理碰撞 + 多层导航

Map 对外提供的接口是统一的，但内部实现完全取决于业务：

寻路的复杂度不该由“算法多酷”决定，而该由“战斗机制需要什么”决定。

预设路径型的简单 SLG 硬上 NavMesh，反而会让关卡配置变难、寻路开销爆炸。

职责四：范围查询

战斗里最高频的查询之一：

“以这个位置为中心，半径 R 内有哪些敌人？”

如果走 EntityManager 直接遍历过滤是 O(n)；Map 可以做空间索引来加速：

要不要做索引，看战斗规模：

  • 单位 < 100：暴力遍历就够，不需要索引
  • 单位 100~1000：网格哈希通常够用
  • 单位 > 1000：需要更精细的空间索引（四叉树 / KD-Tree）

但接口对调用者透明——业务代码只关心“我要查这个圆里的敌人”，Map 内部是遍历还是查索引，是它自己的事。

不同战斗类型，Map 的样子完全不同

把上面四件事拼在一起，会发现：Map 没有“标准答案”——它的体量完全由战斗机制决定。

战斗类型 Map 主要承担 体量
卡牌战斗 抽象阵位（前排/后排槽位），无空间概念 极小（甚至不需要）
回合制 SLG 离散阵位 + 预设路径 + 边界 薄
即时 SLG 连续 xy + 边界 + 范围查询 + 简单寻路 中
RTS NavMesh + 实时寻路 + 视野 + 高频空间查询 厚
3D 沙盒 / 开放世界 3D 空间 + 物理 + 寻路 + 视野 极厚

这种差异不是设计水平的高下，而是业务复杂度本身的差异。

举几个常见的反例：

  • 50 单位的小战斗硬上 NavMesh —— 维护成本爆炸，收益微弱
  • 阵位制卡牌战斗硬上连续坐标 + 寻路 —— 策划反而难以配置关卡
  • 2D 战斗硬上 3D 坐标系 —— 浮点精度问题翻倍，确定性维护成本上升

所以 Map 设计上还有一条更上层的原则：

Map 的复杂度，不应该由“引擎能做什么”决定，而应该由“战斗机制实际需要什么”决定。

结语

到这里 Map 的故事就讲完了——它解决的是实体的空间维度。它要回答的问题是固定的几类：

  • 出生位置（单位、召唤物、掉落物在哪出现）
  • 空间边界（移动、子弹、区域的限制）
  • 寻路（从 A 到 B 怎么走）
  • 范围查询（以某点为中心的圆 / 矩形 / 锥形里有哪些目标）

但每一类问题在不同战斗里的实现复杂度差异巨大。Map 没有标准答案，只有“业务决定的答案”。

跨战斗类型有一条共通的设计原则：

逻辑层和表现层的坐标系必须分开——逻辑层只承载战斗机制需要的维度，表现层负责视觉表达。

到目前为止，逻辑层已经回答了三个问题：

  • 实体本身（EntityManager）
  • 实体之间的关系（Camp）
  • 实体的空间维度（Map）

但战斗里还有一类东西没碰：所有“随机”的行为——暴击、闪避、掉落、AI 选目标——它们到底是怎么保证可复现的？
