沙盒类SLG战斗(三十一) 战斗模式：Controller 统一结构，Mode 隔离业务

战斗跑法的多样性

战斗系统要支持多种“跑法”：

  • 实时战斗：逻辑层和表现层同时推进
  • 回放：逻辑层已跑完，表现层从 Record 数据里播放
  • 编队编辑：只显示布阵界面，不跑战斗
  • 先布阵再回放：先进编队编辑，确认后播放服务器预先算好的结果
  • 先布阵再实时：先进编队编辑，确认后实时跑战斗

本质问题不是“有多少种跑法”，而是这些跑法的大部分流程是相同的——地图加载、单位创建、特效播放——只有少数环节不同。如果每种跑法各写一套，大量代码重复，改一处要改好几处。

Controller：合并相同流程，继承覆写不同流程

和逻辑层的 Processor 思路一样——先由各个模块对外提供能力（地图加载、编队初始化、实时战斗推进、回放阶段控制、特效等），再由 BattleController 来编排这些能力、定义流程骨架。不同跑法继承 Controller，只覆写真正不同的部分，公共能力直接复用：

公共模块在基类统一初始化和销毁，子类只处理自己不同的那部分。相同的流程合并在基类，不同的流程在子类里覆写——新增一种跑法，继承一个新 Controller 就行，已有代码不动。 这本质上是一个「模板方法（Template Method）」的应用。

Mode：把业务关切隔离在战斗之外

Controller 解决了“战斗怎么跑”，但还有另一层问题：同一种跑法，在不同玩法场景下的业务行为不同。

比如同样是“先布阵再实时战斗”：

  • 竞技场：战斗开始时打开竞技场 UI，结束时发送竞技场结算协议
  • 战役：战斗开始前播放剧情动画，胜利时解锁下一关

这些是业务逻辑，不属于战斗系统内部。我们希望战斗模块和业务彼此隔离——战斗系统只维护战斗逻辑，业务方只维护自己的代码，互不干扰，降低双方的维护成本。如果把业务直接写进 Controller，每次新增玩法都要改 Controller，战斗系统和业务的边界就模糊了，谁改谁都要小心翼翼。

做法是在 Controller 里持有一个 Mode，在战斗生命周期的关键节点调用 Mode 的钩子，业务逻辑完全由 Mode 自己处理：

战斗系统不需要知道外面是竞技场还是战役——它只管在对的时机调 Mode 的钩子，剩下的事情 Mode 自己负责。战斗内部对业务无感知，业务对战斗内部无干扰。

新增玩法的正确姿势

Mode 的设计目标是：加一种新玩法，只需要新建一个 Mode 类，逻辑全部收拢在 Mode 内部，不去好几个地方注册，不修改任何已有代码。

在玩法入口把 Controller 和 Mode 组合：

新人接手时，找到 Mode 基类，继承，覆写关心的生命周期，完成——不需要理解整个战斗系统的内部细节，也不会因为改错地方搞坏已有玩法。

结语

战斗模式的设计分两层：Controller 解决“怎么跑”，Mode 解决“业务在哪介入”。

Controller 用继承把不同跑法的差异收拢在各自子类，公共流程在基类合并；Mode 用钩子把业务逻辑隔离在战斗之外，战斗内部对玩法完全无感知。两层各司其职——新增跑法加 Controller 子类，新增玩法加 Mode 子类，扩展点清晰，逻辑不散。

到这里，表现层的五章全部结束。从 Record 消费到实体组件，从插值到弹道特效，再到战斗模式——表现层的每个模块都在做同一件事：把逻辑层的离散数据，演绎成玩家看到的连续画面。——逻辑决定结果，表现决定体验。
