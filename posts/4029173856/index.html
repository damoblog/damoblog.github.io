沙盒类SLG战斗(十三) 整体运转：Bridge 如何驱动 Engine

点我回到首页

从一个朴素的疑问开始

第三章和第四章曾经从整体视角讲过逻辑层是怎么被驱动的——同帧收敛、输入→执行→结算→输出→判定。但那时候各个模块还没有展开，读者对”输入怎么收””输出怎么发””执行顺序谁定”这些问题只有一个轮廓。

现在所有模块都讲完了。Input 怎么缓冲排序、Record 怎么按帧收集、Processor 怎么锁定执行顺序、Config 和 Random 怎么保证双端一致——这些细节都已经落地。

可以回到一个更完整的问题了：

这些模块组装好以后，整个战斗系统到底是怎么跑起来的？

“谁来告诉 Engine 现在该 tick 了”？”每隔多久 tick 一次”？”输入从哪里灌进来、结果往哪里送”？

这些问题的答案不在逻辑层里，而在逻辑层的外面——桥接层。

逻辑层是一台无状态的计算机

在讲桥接层之前，先说清楚逻辑层本身是什么。

逻辑层是一个纯粹的确定性计算引擎——它不拥有时间，不携带业务，不关心自己在哪里运行。

它不知道自己是在客户端跑还是在服务器跑，不知道这场战斗发生在什么场景、什么模式下，不知道”XX 场景下需要给玩家加什么 Buff”——这些业务判断全部发生在外面。逻辑层只接收外部传入的数据：初始阵容、配置表、随机种子、每帧的输入——然后按固定规则算，输出结果。

甚至连 tickInterval（每帧的逻辑时长）都是从外部传入的，不是逻辑层自己决定的。Engine 内部只做一件事：每次被 tick 时，把 tickDuration 累加一个 tickInterval——但这个间隔是多少、多久 tick 一次，都不归它管。

这个设计是刻意的。逻辑层越”无知”，它就越纯粹：

  • 不掺杂业务 → 不同玩法模式可以复用同一套引擎
  • 不拥有时间 → 服务器和客户端可以用完全不同的节奏驱动它
  • 不依赖环境 → 单元测试、自动化仿真、编辑器预览都能直接跑

逻辑层不是一个”应用程序”，而是一台被外部驱动的计算机——喂什么数据就算什么，不问为什么。

桥接层的中心：BattleBridgeCore

那”驱动”这件事由谁来做？

答案是桥接层。第二章讲过，桥接层负责逻辑层和外部世界之间的翻译。但更准确地说，桥接层的核心——BattleBridgeCore——是逻辑层对外暴露的唯一接口。

它包住了 Engine，只向外暴露几个方法：

外部——无论是客户端还是服务器——只能通过 BattleBridgeCore 和逻辑层交互。不能直接拿到 Engine，不能直接修改任何 Manager 的状态，不能绕过 Input 往里塞东西。

这不是出于”代码整洁”的考虑，而是确定性的要求：一旦外部能绕过桥接层直接碰 Engine，Input/Record 的收敛机制就可能被绕过，确定性就无从保证。

BattleBridgeCore 不只是封装，它是确定性边界的守门人。

同一个引擎，两种驱动方式

有了这个统一接口，不同的使用方就能用完全不同的方式驱动同一个引擎。

服务器：一口气跑完

服务器不需要表现——它只需要结果。所以它的驱动方式极其简单：一个 while true 循环，每圈 tick 一次，直到战斗结束：

没有等待，没有渲染，没有表现层。一场可能持续几分钟的战斗，服务器可能几百毫秒就跑完了。这就是第十章里”服务器校验”的完整流程——拿到客户端上报的初始状态和输入序列，一口气重跑一遍，比对结果。

客户端：按时间节奏驱动

客户端需要表现。玩家要看到单位移动、技能释放、血条变化——这些都需要时间。所以客户端不能一口气跑完，而是每隔一个逻辑帧间隔（比如 0.125 秒）才 tick 一次：

每帧 Update 被渲染循环调用，累加 dt。攒够 0.125 秒，才触发一次逻辑 tick。逻辑 tick 内部和服务器完全一样——调 TickBattle，走 Processor 流程。但 tick 之后多了一步：取出这帧的 Record，交给表现层播动画。

两种驱动方式的差别只在**”多久 tick 一次”和”tick 完了干什么”**——Engine 内部跑的是完全相同的逻辑。

回放：第三种驱动方式

除了正常对局和服务器校验，还有第三种场景——回放。

回放介于两者之间：和客户端一样按时间节奏推进（玩家要看画面），但不需要跑逻辑——因为 Record 已经录好了。

不需要 Engine，不需要 BattleBridgeCore，甚至不需要逻辑层——只需要一条 Record 时间线和一个表现层。这也从另一个角度印证了第十一章的结论：Record 是为表现服务的，不是为逻辑复现服务的。

一个被低估的事实

回头看这三种驱动方式：

服务器 客户端 回放
跑逻辑 ✓ ✓ ✗
有表现 ✗ ✓ ✓
时间控制 全速 按帧率 按帧率

同一个 Engine，在服务器上是一台验证机器，在客户端上是一台带画面的模拟器，在回放里甚至不需要出场。这之所以可能，是因为前面十二章一直在做同一件事：把逻辑层设计成一个不依赖任何外部环境的纯计算单元。

它不拥有时间（由外部决定何时 tick）、不携带业务（由外部决定灌什么数据）、不依赖表现（由 Record 解耦）——所以它可以被任何方式驱动，在任何环境里运行。

Bridge 不只是”逻辑和表现之间的翻译层”——它决定了 Engine 以什么节奏、在什么环境、为什么目的运转。同一台引擎，不同的 Bridge，就是不同的使用场景。

结语

这一章把视角从逻辑层内部拉了出来，看的是”逻辑层是怎么被外部驱动起来的”。

逻辑层本身是一台无状态的计算机，不关心时间、不关心业务、不关心环境。桥接层通过 BattleBridgeCore 包住它，决定什么时候 tick、灌什么数据、拿走什么结果。服务器一口气跑完拿结果，客户端按帧率推进配表现，回放甚至不需要逻辑层——三种场景，同一套引擎。

这就是前面十二章所有设计决策的回报：一个足够纯粹的引擎，才能被足够灵活地使用。

到这里，战斗系统的大框架就全部讲完了——三层结构、模块拆分、通信机制、驱动方式。但框架终究只是骨架，真正让战斗”有内容”的，是骨架里跑的东西：实体本身是怎么设计的？

一个单位怎么决定自己该做什么？技能、Buff、AI、属性这些能力怎么组织在一起？状态之间怎么切换？——这些才是战斗系统里最核心、也最复杂的部分。

下一章，视角从系统框架回到最底层——实体的内部结构。
