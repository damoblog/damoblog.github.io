沙盒类SLG战斗(二十三) 技能系统（四）：标签系统与自定义属性

还没回答的两个问题

上一章用 Validator 解决了条件判断，用 Calculator 解决了数值计算。但实际项目中，策划很快会提出新的需求：

  • “这个技能对亡灵造成额外伤害，对人类降低防御”——亡灵、人类这些种族定义，写在哪？
  • “这个伤害是魔法伤害，应该被魔法抗性减免，而不是物理抗性”——魔法伤害、物理伤害这种分类，写在哪？
  • “每攻击 3 次触发一次额外效果”——这个计数器写在哪？

游戏里充满了这类”枚举型定义”——种族（亡灵、人类、精灵）、阵营（魏、蜀、吴）、兵种（步兵、骑兵、弓兵）、伤害类型（物理、魔法）、技能分类（普攻、技能、Boss 技能）……以及各种需要在战斗过程中动态维护的状态和计数。

这些东西不是属性（攻防血），不是 Behavior（做什么），也不是 Validator（判断什么）——它们需要自己的表达方式。

如何支持枚举型定义

先看第一个问题。”亡灵”、”人类”这些分类，本质上是一种枚举型定义。最直觉的做法是为每种分类维度定义一套枚举，然后利用上一章的 Validator 体系，为每种枚举写一个对应的验证器（种族验证器、阵营验证器、兵种验证器……）。但问题是：枚举会无限膨胀，验证器也会跟着膨胀。

种族是一套枚举，阵营是一套枚举，兵种是一套枚举，设定（Boss、精英、小怪）是一套枚举……每加一种分类维度，代码里就多一套 enum。更麻烦的是，这些分类之间经常需要交叉判断——“目标是亡灵 且 是步兵”——硬编码的枚举之间很难组合。

这里我们引入一个新的概念——标签（Tag）。不按维度定义枚举，而是给每个对象打标签——一个标签就是一个数字 ID，对象可以同时拥有多个标签。

用标签替代：不按维度分，所有分类都是同一个体系里的数字 ID。一个单位可以同时拥有多个标签：

种族、阵营、兵种、设定——全部统一成标签。不再需要一个维度一套枚举，加一个新分类只是加一个新的标签 ID。策划可以像配表一样自己新增标签，然后填充到对应的单位身上——不需要程序改代码、不需要加枚举。

标签的判断通过标签验证器来做：

标签验证器本身也是一个 Validator——它可以和属性验证器、逻辑验证器自由组合。”目标是亡灵 且 HP 低于 50%”就是一个 AND 逻辑验证器，里面包含一个标签验证器（亡灵）和一个属性验证器（HP < 50%）。

标签不只用在单位上——技能也有标签，伤害也有标签。比如策划需要”减少所有普攻伤害 20%”，就在伤害计算时检查技能标签里有没有”普攻”标签。”免疫所有魔法伤害”，就在受伤时检查伤害标签里有没有”魔法伤害”标签。整个系统里只有一种标签机制，但它可以标记在任何对象上。

标签系统用一套统一的 ID 体系，替代了无数个硬编码的枚举。枚举会膨胀，标签不会——因为加一个新标签只是加一行配置，不需要改任何代码。

如何让计数、状态可配置

再看第二个问题。战斗中经常需要一些动态的、临时的数据——不是角色的基础属性（攻防血），而是技能机制需要的状态：

  • “每攻击 3 次触发一次额外效果”——需要一个攻击计数器
  • “受到致命伤害时，如果有剩余复活次数，则不死亡”——需要一个复活次数
  • “进入狂暴状态后，攻击速度提升”——需要一个狂暴标记

这些数据写在哪？

写在单位的基础属性里？不合适——它们是某个技能或机制专用的，不是所有单位都有的通用属性。往基础属性表里加”攻击计数””复活次数””狂暴标记”，属性枚举会无限膨胀，和上面标签的问题一样。

写在 Behavior 代码里？那这个状态就被 Behavior 私有了——其他 Behavior 读不到，Validator 也判断不了，Calculator 也引用不了。

和标签的思路类似，我们可以在属性系统里开辟一块动态区域——叫做自定义属性——允许技能在运行时创建和修改自定义的属性值。

自定义属性的 ID 和普通属性一样，是一个数字——但它不在基础属性枚举里，而是由技能配置动态定义的。和标签一样，策划可以自己按需新增和使用，不需要程序改代码。它和普通属性最大的区别是：普通属性是所有单位共有的，自定义属性是按需创建的。

自定义属性的价值在于它能被整个系统消费：

  • Validator 能读它：判断”攻击计数 >= 3”，用属性验证器就行
  • Calculator 能引用它：伤害公式里加上”复活次数 × 系数”，用 Calculator 配置就行
  • Behavior 能改它：攻击命中后用一个 PropertyModifier 给计数器 +1，计数器满了再重置为 0

用”每攻击 3 次触发特殊效果”来看完整的配置流程：

开局释放被动技能，注册两条链：第一条监听攻击事件，每次攻击命中时给自定义属性 +1；第二条监听自定义属性变化事件，当属性值到达 3 时触发特殊效果。整个过程没有写一行代码——全部通过已有的 Behavior + Validator + 自定义属性组合完成。

自定义属性把”技能机制需要的临时数据”从代码里解放出来，变成了可配置的、可被系统消费的数据。它不是一个新系统，而是属性系统的延伸——让属性不再局限于”攻防血”这些固定维度。

结语

这一章补完了技能系统最后的两块拼图。

标签系统用一套统一的 ID 替代了无数个硬编码的枚举——种族、阵营、兵种、伤害类型、技能分类，全部是标签。自定义属性让技能可以在运行时创建和使用临时数据——计数器、状态标记、复活次数，不需要往基础属性表里硬塞。

到这里，技能系统的各个机制都已经讲完了。但光知道每个模块的职责还不够——它们组合在一起的时候，整个链路是怎么跑的？下一章会拿一个复杂的技能，从头到尾走一遍完整的拆解和执行流程，把这几章的内容融汇贯通。
