沙盒类SLG战斗(十四) 实体的内部结构：组件化设计

系统里被运转的东西，长什么样

前面十三章讲的都是”系统怎么运转”——三层结构、模块拆分、通信机制、驱动方式。但系统里真正被运转的东西，一直没有展开过：实体本身是怎么设计的。

一个战斗单位要做的事非常多：要有属性（攻防血），要有技能（主动技、被动技），要有 AI（选目标、做决策），要有状态机（待机、移动、攻击、死亡），要有 Attachment 系统（战斗内的各种效果）、寻路、触发器……

这些能力怎么组织在一个实体里？一个实体每帧被 tick 的时候，内部这些能力按什么顺序执行？不同类型的实体（近战兵、远程兵、指挥官、召唤物）怎么共享通用逻辑又保持各自的差异？

这一章就回答这些问题。

为什么是组件化

最朴素的想法是继承——定义一个 BaseUnit，把属性、技能、AI 全写在里面，然后不同单位类型各自继承、覆写需要的部分。

这套在早期能跑，但一旦单位类型多起来、能力组合变得复杂，问题就会暴露：

  • 近战兵有寻路但没有弹道，远程兵有弹道但寻路逻辑不同，召唤物有 AI 但不需要技能 CD——这些差异用继承来表达，很快会变成一棵庞大的继承树，牵一发动全身
  • 想给某类单位加一个能力（比如护盾），要么改基类（影响所有人），要么在子类里硬塞（到处重复）

根本原因是：**继承是”我是什么”的抽象，但实体的能力更像”我有什么”**——一个单位”有”属性、”有”技能、”有”AI，这些能力是可以自由组合的，不是固定的层级关系。

组件化就是顺着这个思路走：把每种能力封装成独立的组件，实体本身只是一个容器，按需组装。这个思路其实和 Unity 的 MonoBehaviour 设计原则一脉相承——GameObject 本身是空的，所有能力都通过挂载组件来获得。我们在逻辑层里做的是同样的事，只是组件不再是引擎提供的渲染、碰撞，而是战斗逻辑里的属性、技能、AI。

不同类型的实体，就是不同的组件组合：

值得一提的是，Entity 本身只是一个”容器 + 组件生命周期管理”的基类——它不包含任何战斗逻辑，只负责持有组件、驱动组件的初始化和 tick。这意味着不只是战斗单位可以继承它，任何需要组件化能力的对象都可以。比如第七章讲过的 Camp（阵营），它本身也可以继承 Entity，挂载阵营级别的组件（胜负判定、阵营属性、阵营 Attachment），复用同一套生命周期和组件管理机制。

组件化的核心不是”代码复用”，而是”能力可组合”——实体的差异通过组件的增减和替换来表达，而不是通过继承层级。

两层多态

组件化带来了一个自然的结构：实体本身可以有多态，组件也可以有多态。

实体层多态：不同类型的实体（近战兵、远程兵、指挥官、召唤物）决定自己需要哪些组件、使用哪种组件实现。它们都是”实体”，但组装方式不同。

组件层多态：同一种组件在不同实体类型下可以有不同的实现。但这里的多态往往不是整个组件完全重写，而是部分函数的覆写——基类组件实现了通用逻辑（生命周期管理、数据存储、通用接口），子类只覆写需要差异化的那几个方法。比如 AI 组件的基类处理了目标缓存、搜索流程这些通用逻辑，子类只覆写”搜索策略”和”施法决策”：近战兵是”找最近的敌人打”，指挥官是”站在后排释放全局技能”，召唤物是”跟随主人移动”。

同时，组件本身是可复用的——不是每种实体都需要一套全新的组件。很多组件在不同实体类型之间完全共用，只有确实存在行为差异的地方才需要派生子类：

这两层多态是正交的——实体决定”组装什么”，组件决定”具体怎么做”。大部分组件直接复用基类，只有少数需要差异化的组件才派生子类覆写个别方法。加一个新的单位类型，往往只需要写一两个新的组件子类，其余直接复用。

一帧之内：不亏帧，不多帧

组件解决了”能力怎么组织”的问题，但还有一个更细的问题：一个实体被 tick 的时候，内部这些组件按什么顺序执行？

这个问题比看起来要微妙。考虑一个具体的例子：

一个单位这帧要做两件事——减少技能 CD，然后释放技能（如果 CD 好了的话）。

如果先减 CD、再判断释放：这帧减完 CD 恰好归零，立刻就释放了——等于这帧既减了 CD 又执行了技能的第一帧，一帧干了两帧的事。

如果先判断释放、再减 CD：CD 还没减完，判断为”不能释放”，然后才减 CD——等于这帧白白浪费了一次判断机会，CD 归零了但要等下一帧才能释放。

两种顺序都不对。核心原则是：一帧的产出必须恰好是一帧的量——不亏帧，不多帧。

解法是把一帧分成三个阶段，以状态机（或 AI 决策）为分界线：

前段：收集信息、更新依赖

  • 技能 CD 减少
  • AI 搜索目标、评估战场
  • 处理好这一帧决策所需要的所有前置数据

中段：决策

  • 状态机评估当前状态，决定”这一帧要做什么”
  • 这是整帧的分界线——前面的数据准备完了，后面的执行还没开始

后段：执行与结算

  • 技能施法标记（标记”要施法了”，实际技能逻辑下一帧才开始）
  • 正在执行中的技能推进（已经在施法过程中的技能继续走帧）
  • Attachment 到期检查
  • 属性变化统一生效（延迟结算）

这其实就是第三章讲过的延迟结算在实体内部的落地。第三章说的是系统级别的延迟结算——一帧内产生的数值变化先记录、帧末统一生效，防止执行顺序污染结果。这里是同一个原则下沉到了实体级别：技能 CD 减少、Buff 增减这些操作在各自的时机直接执行，但它们引起的属性变化不会立刻生效——属性组件在中间过程中只打脏标记，到帧末统一重算一次。这样做既避免了一帧内属性被反复重算的性能浪费，也保证了同一帧内所有决策看到的属性是同一份快照。

从数据流的角度看，三段分别对应读取状态 → 做出决策 → 写回结果。前段为决策准备数据（比如 CD 减完了），中段立刻就能看到并做出判断（决定施法），但技能的第一帧要到下一帧才真正执行——这帧只标记”要施法了”，不会在同一帧里既减 CD 又跑技能逻辑。

这里可能会有一个直觉上的疑问：”决策和执行隔了一帧，手感不会延迟吗？”在逻辑层面确实存在一帧的间隔，但表现层可以在收到”将要释放技能”的记录时提前开始动画预表现，从而掩盖逻辑帧的延迟——玩家感知到的响应是即时的。

三段式不是随意划的——它确保”读””判””写”不会在同一帧里互相影响，从而保证每帧的产出量恰好是一帧。这个模型不只适用于战斗实体，任何需要确定性逐帧推进的系统都可以套用。

谁来决定组件的执行顺序

到这里有一个设计上的分歧，值得展开说。

一种做法是组件自治——每个组件自己决定在 PreTick、Tick、LateTick 哪个阶段执行，实体只是依次调用。这种写法最自然：组件添加时自动注册到对应的阶段列表里，执行时按列表顺序走。

这套能跑，但随着组件变多、组件之间的依赖变复杂，会出现一个问题：组件之间的执行顺序变得隐式且脆弱。

比如：Attachment 组件要在 AI 组件之前执行（因为 Attachment 可能影响 AI 决策），技能组件要在状态机之后执行（因为状态机决定能不能施法）。这些依赖关系藏在”谁先注册到 preTickComponents 列表里”这种隐式顺序中——加一个新组件时，很难知道它应该排在哪里。

更麻烦的是跨阶段依赖：如果 A 组件的 PreTick 要在 B 组件的 PreTick 之前，但 A 组件的 Tick 又要在 B 组件的 Tick 之后——这种交叉顺序在”每个组件自己注册到对应阶段”的模型里几乎无法干净地表达。

另一种做法是实体编排——回到第五章 Processor 的思路：组件只提供能力（”我能减 CD””我能做 AI 决策”），实体来决定调用顺序。

这种写法的好处和 Processor 一样：所有组件的执行顺序在一个地方显式定义，依赖关系一目了然。 加一个新组件时，在这个列表里找到它应该在的位置插进去就行——不用猜”它注册到哪个阶段列表的第几位”。

组件自治解决的是”各管各的”，但战斗里组件之间天然有依赖。谁先谁后不能散落在各处——应该由一个地方统一说了算。

这两种做法在实际项目里都能见到。我经历过组件自治的方案，能跑但维护成本会随组件数量上升。如果重新设计，我会选择实体编排——这和第五章 Processor 统一调度 Manager 是同一个设计原则，只是粒度从”Engine 级别”下沉到了”Entity 级别”：执行顺序是架构决策，不是实现细节，它应该被显式定义在一个地方，而不是分散在各个组件的注册逻辑里。

结语

这一章把视角从系统框架拉回了最底层——实体的内部结构。

组件化解决了”能力怎么组织”：每种能力一个组件，实体是容器，按需组装。两层多态让实体层和组件层可以独立变化。”不亏帧不多帧”的原则把一帧切成了决策和执行两个阶段，确保每帧的产出量恰到好处。而组件的执行顺序，和 Processor 调度 Manager 一样，应该由上层统一编排，而不是各自为政。

框架讲完了，实体的骨架也讲完了——接下来会逐个展开骨架上的组件：属性、Attachment、AI、状态机，最后再到最复杂的技能系统。先从最基础的开始。
