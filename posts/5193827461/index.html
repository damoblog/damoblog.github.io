还没回答的两个问题

上一章确立了技能系统的核心认识：系统的最小单位是行为（Behavior），而不是技能。 Filter 负责收集和筛选目标，OnHit 负责消费目标，两者统一为同一个接口。

但有两个问题还没回答：

  1. 多个 Behavior 怎么串起来？一个技能里的 Filter → OnHit → OnHit 这条链是怎么跑的？
  2. 技能在什么时机触发这些 Behavior？是释放瞬间一口气全跑完，还是有别的模式？

这一章就回答这两个问题。

BehaviorChain：Behavior 的容器

上一章说”技能是若干 Behavior 的组合”，但组合本身也需要一个载体——不能让技能自己去遍历一堆 Behavior ID。

这个载体就是 BehaviorChain。它持有一组 Behavior ID 的列表，运行时按顺序依次执行：

关键设计：BehaviorChain 本身也实现了 ISkillBehavior 接口。 这意味着在任何需要 Behavior 的地方，都可以放一个 BehaviorChain——它和单个 Behavior 在调用方看来没有区别。技能配置里写的不是”先跑哪个 Filter、再跑哪个 OnHit”，而是”跑这个 BehaviorChain”，链内部的编排对外部透明。

换句话说，BehaviorChain 不是一次性的执行列表，而是一个可复用的技能片段——它可以被不同的技能引用、在不同的时机触发，就像一个独立的配置资产。

还有一个细节值得注意：链是短路的——任何一个 Behavior 返回 false，后续的 Behavior 就不再执行。这很自然：如果 Filter 没找到任何目标，后面的 OnHit 就没有意义了。

targets 在链上怎么流转

BehaviorChain 里的所有 Behavior 共享同一个 targets 列表。这个列表像流水线上的工件一样，依次经过每个 Behavior 的加工。

注意几个要点：

  • Filter 类 Behavior 改变 targets（往里加、往外筛），OnHit 类 Behavior 消费 targets（对列表中的每个目标施加效果，列表本身通常不变）
  • 中间如果某个目标已死亡或不满足条件，OnHit 会跳过该目标并将其从列表中移除——这样后续 Behavior 就不会再处理它
  • 整条链共享同一个 targets 引用，前一步的输出就是下一步的输入，不需要任何额外的数据传递机制

targets 是链路中的唯一数据载体，所有 Behavior 都只围绕它读写——不需要额外的上下文系统，不需要 Behavior 之间互相传参。

BehaviorChain 不是 Behavior 的管理者，而是 Behavior 的流水线——targets 在链上流转，每一站加工一次。

技能是一条时间轴

有了 BehaviorChain，再来看技能本身。

一个主动技能从释放到结束，本质上是一条时间轴——它有明确的开始时间、持续时长，以及在某些时间点触发特定行为的关键帧（KeyFrame）：

每到一个关键帧，技能就触发该帧上配置的一个或多个 BehaviorChain。Tick 的实现很直接：

每一帧检查有没有关键帧，有就逐个运行上面挂的 Behavior（通常是 BehaviorChain）。这就是技能的运转核心——时间轴驱动关键帧，关键帧触发 BehaviorChain，BehaviorChain 驱动 Behavior 流水线。

当然也有一些技能不需要时间轴——比如被动技能，在战斗开始时直接运行所有 Behavior，没有帧的概念：

被动技能的场景比较简单，不展开。重点在于：有时间轴的主动技能才是技能系统复杂度的主要来源——因为它引出了一个核心问题：Behavior 在什么时机被触发？

几种典型的触发时机

到目前为止，我们只看到了一种触发方式：关键帧直接执行——时间轴走到某一帧，立刻触发 BehaviorChain。

但实际战斗中远不止这一种。这里列几种典型的触发时机——不是全部，但覆盖了最常见的场景。

直接执行

最简单的方式：技能走到某一帧，直接触发 BehaviorChain。

适用场景：近战攻击、瞬发技能——效果在施法的某一帧立刻生效，不需要任何中间过程。

通过载体执行

远程攻击不是”施法即命中”的——弹道需要飞行时间。

这种情况下，关键帧触发的不是伤害，而是发射一枚弹道。弹道携带着”命中后要执行的 BehaviorChain”在战场上飞行，到达目标后再触发：

弹道就是 Behavior 的载体——它把”执行效果”这件事从”施法帧”延迟到了”命中帧”。对 BehaviorChain 来说，自己在什么时候被执行是透明的——不管是关键帧直接调用，还是弹道命中后调用，链的内部逻辑完全一样。

通过 Attachment 持续执行

有些效果不是一次性的，而是持续生效的——比如持续掉血、区域灼烧。

这种情况下，技能不是直接造成伤害，而是给目标挂一个 Attachment，这个 Attachment 会在自己的生命周期内周期性地触发 BehaviorChain：

这样就把 Behavior 的触发从”某一帧”变成了”一段时间内周期性触发“。这和第十六章讲的 Attachment 系统自然地结合在一起——Attachment 管理生命周期，Behavior 提供执行逻辑。

通过事件触发

还有一种常见的时机：当某个战斗事件发生时触发。

比如”每次受到伤害时反弹 10% 伤害”、”攻击时有概率施加眩晕”——这些效果不是按时间触发的，而是按事件触发的。

实现方式类似——技能给目标挂上一个带触发器的 Attachment。这个 Attachment 注册到触发器系统上，当特定事件发生时执行对应的 BehaviorChain：

这就是第十八章讲的触发器系统和技能系统结合的具体方式——触发器提供”事件 → 响应”的能力，Attachment 提供生命周期管理，Behavior 提供具体的执行逻辑。三者组合起来，就是”在一段时间内，每当某事发生时执行某个行为”。

触发时机的统一视角

整理一下这几种触发时机：

从 BehaviorChain 的角度看，这些时机没有本质区别——链本身的执行逻辑完全一样，差异只在于”谁在什么时候调用了我的 TakeEffect“。这就是上一章说的统一接口的好处：同一组 Behavior，直接执行能用、挂在弹道上能用、放进 Attachment 里也能用——行为是可复用的，触发时机是可配置的。

实际游戏中的触发时机远不止上面列的这些——Attachment 被驱散时、单位出生时、单位销毁时、技能施法开始时、技能施法结束时……每个项目的需求不同，触发时机的种类也会不断扩展。但扩展的方式是统一的：定义一个新的触发条件，在对应的逻辑节点调用 BehaviorChain 就行了——Behavior 本身不需要任何改动。

技能系统的两个核心维度：Behavior 决定”做什么”，触发时机决定”什么时候做”。两者完全解耦，因此同一套行为可以在任意时机复用。

结语

这一章补完了技能系统的运转机制。

BehaviorChain 把多个 Behavior 串成流水线，targets 在链上逐站流转。技能本身是一条时间轴，通过关键帧在特定时间点触发 BehaviorChain。而 Behavior 的触发不只有”关键帧直接执行”一种——弹道载体、持续效果、事件触发等多种时机覆盖了战斗中绝大部分场景，且对 BehaviorChain 本身完全透明。

到这里，技能系统的骨架已经立起来了：Behavior 是原子、BehaviorChain 是组合、时间轴是驱动、触发时机是调度。 但还有几个配套问题没有展开：条件判断怎么做？数值计算怎么可配置？
