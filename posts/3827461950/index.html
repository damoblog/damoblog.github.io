沙盒类SLG战斗(二十八) 表现层的实体与组件：逻辑层的镜像

组件化是一种设计理念，不只属于逻辑层

第十四章给逻辑层的实体做了组件化设计。组件化本身是一种设计理念——把一个复杂对象拆分成多个独立模块，每个模块只管自己的关注点，互不干扰。这个理念不属于哪一层，逻辑层可以用，表现层同样可以用。

上一章讲了 Record 怎么被消费。但 Record 里说“单位 A 的 HP 变了”，表现层必须先有一个“单位 A”——那这个“单位 A”到底是什么？

逻辑层有自己的实体系统，表现层也需要一套。两者的职责完全不同：

  • 逻辑层的实体管的是数据和规则——HP 是多少、攻击力是多少、技能 CD 到了没
  • 表现层的实体管的是渲染和演出——模型放在哪、动画播什么、血条显示多少

一个单位在两层各有一个“实体”，逻辑层的那个负责算，表现层的那个负责演。Record 是两者之间唯一的通信通道。

表现层的组件

表现层实体同样采用组件化——一个单位的渲染表现，由若干职责单一的组件拼起来，每个组件只管自己那一块，互不依赖。组件的类型和逻辑层完全不同，但设计理念是一样的。

ViewComponent：管位置和朝向。它持有模型的 Transform，负责把逻辑坐标转成渲染坐标，驱动模型在屏幕上的移动和旋转。这个组件是最复杂的——因为逻辑帧和渲染帧的频率不同，中间需要插值（下一章详细讲）。

HpComponent：管血条。收到 HpChangeRecord 时，它不是直接把血条设到目标值，而是记录起始值和目标值，然后在后续的渲染帧里平滑过渡。血条“滑下去”的动画就是这个组件在做。

StateComponent：管动画状态。它有一个简单的状态机——待机、移动、攻击、死亡。收到对应的 Record 时切换状态，状态变化时触发事件通知动画系统播放对应的动画。

HeadComponent：管头顶 UI——名字、等级标签、状态图标。相对简单，跟着 ViewComponent 的位置走就行。

Record 怎么找到对应的表现层实体

Record 里说“battleId=5 的单位 HP 变了”，表现层怎么把这条指令送到正确的实体上？

表现层维护一个实体注册表——一张以 battleId 为 key、以 ClientArmy 为 value 的映射表。单位出生时注册进去，死亡时移除。每条需要作用到具体单位的 RecordElement，在 PlayElement 里先查这张表拿到实体，再调对应组件的方法：

调度链是：TickData → RecordElement.PlayElement → 查实体注册表 → 调组件方法。

这里有几个细节值得注意：

Record 直接指名道姓调组件：actor.hpComponent:SetHp(...) 而不是 actor:OnRecord(record)。这意味着 RecordElement 知道自己要驱动哪个组件——设计上是有意为之的。HpChangeRecord 天然就对应 HpComponent，让 RecordElement 自己持有这层知识，比写一个集中的分发器更直接，也更容易追踪。

找不到实体就跳过：if actor == nil then return end 是必要的防御，而不是可选的。录像模式下数据来自外部，实体有没有创建、是否已经销毁，都不在 Record 的控制范围内。跳过比崩溃好。

不是所有 Record 都需要查实体：全局性的 Record（比如全场天气变化、战场背景切换）直接操作场景，不需要走实体注册表。

本质上，这一步是在把“逻辑层的标识（battleId）”映射成“表现层的对象引用”——Record 携带的是 ID，组件需要的是对象，实体注册表就是这两者之间的桥。

逻辑层和表现层的组件对比

两层的组件用了同样的模式（组件化），但设计思路有一个关键区别：

逻辑层的组件之间有明确的执行顺序——第十四章讲过，属性要在 Attachment 之后重算，技能 CD 要在 AI 决策之前减。组件的执行顺序是架构级的约束。

表现层的组件之间基本没有顺序依赖——ViewComponent 更新位置、HpComponent 更新血条、StateComponent 更新动画，各管各的，互不影响。偶尔有依赖的地方（HeadComponent 需要 ViewComponent 的位置），通过事件通知解决就行，不需要像逻辑层那样严格编排。

这也合理——逻辑层的组件在处理因果关系（先减 CD → 再判断能不能施法 → 再执行技能），表现层的组件在处理并行的视觉元素（模型移动、血条变化、动画播放可以同时进行）。

不同类型的表现层实体

和逻辑层一样，不同类型的单位有不同的组件组合：

上帝实体在表现层没有对应物——它只在逻辑层存在。上帝释放技能产出的 Record（比如全场加攻击力）由对应的 RecordElement 直接作用到各个单位的表现层实体上，不需要一个“上帝的渲染对象”。

这是逻辑层和表现层不对称的一个例子——不是每个逻辑层实体都需要在表现层有镜像。

结语

表现层的实体和组件是逻辑层设计的一面镜子——同样的组件化思路，但完全不同的职责。逻辑层管数据，表现层管渲染；逻辑层组件有严格的执行顺序，表现层组件基本并行；逻辑层实体不一定在表现层有对应物。

组件搭好了，但还有一个核心问题没解决：逻辑帧是 1/8 秒跳一次，渲染帧是 1/60 秒刷一次。逻辑是离散跳变的，画面却必须连续——这中间的断层怎么补？

下一章，插值。
